What is the ORM — and how is it different from Core?

ORM stands for Object-Relational Mapper. The idea is simple: instead of thinking in tables and rows, you think in Python classes and objects. The ORM automatically translates between the two worlds.

You define a class called User with attributes like name and email. The ORM maps that class to a database table called users. When you create a User(name="Alice") object and save it, the ORM writes the INSERT SQL for you. When you query select(User), it reads the rows and hands you back User objects — not raw tuples.

This means you can work entirely in Python objects without ever writing a single line of SQL by hand. The ORM handles all the SQL generation, parameter binding, and result mapping for you.

The golden rule of ORM

In Core, you work with rows and dictionaries. In ORM, you work with Python objects. The object has methods, attributes, and can have business logic. A row from Core has none of that.

Doing the same thing — two ways
Core approach
CORE
# Insert a user
conn.execute(
  insert(users).values(
    name="Alice", age=28
  )
)

# Read users
rows = conn.execute(
  select(users)
).fetchall()
# rows[0] → tuple
print(rows[0][1])  # 'Alice'
VS
ORM approach
ORM
# Insert a user
user = User(name="Alice", age=28)
session.add(user)
session.commit()

# Read users
users = session.execute(
  select(User)
).scalars().all()
# users[0] → User object
print(users[0].name)  # 'Alice'

Tuples, dicts, raw rows. More control, closer to SQL. Better for complex analytics queries.

Python objects with attributes and methods. Relationships. Cleaner code for typical app logic.

Defining Models — Python Classes as Database Tables

A model is a Python class that represents a database table. Every instance of that class represents one row in the table. Every attribute on the class represents one column.

To define models in SQLAlchemy 2.0 (the modern way), your model class inherits from Base, which comes from DeclarativeBase. Each column is declared using Mapped[type] (the Python type) and mapped_column(...) (the column options). This is clean, type-safe, and works perfectly with Python type checkers like mypy.

The complete model definition

Here is the full code for both models — User and Order — the same tables we used throughout Module 04. Notice the clear structure:

1. Base class — created once, inherited by all models.
2. __tablename__ — the exact name of the database table.
3. Mapped[type] — declares the Python type for this column.
4. mapped_column() — sets the database-level options (primary key, nullable, unique, default, etc.).
5. relationship() — links two models in Python only (no column created).

The Optional in Mapped[Optional[str]] (or the int | None syntax) tells SQLAlchemy the column is nullable — it can have no value. Without it, the column is NOT NULL.

models.py — full model definitions
from __future__ import annotations
from typing import List, Optional
from sqlalchemy import String, Float, ForeignKey
from sqlalchemy.orm import (
    DeclarativeBase, Mapped,
    mapped_column, relationship
)

# ONE Base class for the whole project
class Base(DeclarativeBase):
    pass


class User(Base):
    __tablename__ = "users"

    id:    Mapped[int]           = mapped_column(primary_key=True)
    name:  Mapped[str]           = mapped_column(String(100))
    email: Mapped[str]           = mapped_column(String(200), unique=True)
    age:   Mapped[Optional[int]]  = mapped_column(nullable=True)
    city:  Mapped[Optional[str]]  = mapped_column(String(100), nullable=True)

    # Relationship: one User → many Orders (Python only, no column)
    orders: Mapped[List["Order"]] = relationship(back_populates="user")

    def __repr__(self) -> str:
        return f"<User id={self.id} name={self.name!r}>"


class Order(Base):
    __tablename__ = "orders"

    id:      Mapped[int]           = mapped_column(primary_key=True)
    user_id: Mapped[int]           = mapped_column(ForeignKey("users.id"))
    product: Mapped[str]           = mapped_column(String(200))
    amount:  Mapped[float]
    status:  Mapped[str]           = mapped_column(String(50), default="pending")

    # Relationship: one Order → one User
    user: Mapped["User"] = relationship(back_populates="orders")

    def __repr__(self) -> str:
        return f"<Order id={self.id} product={self.product!r}>"
mapped_column() options
Every option you will use inside mapped_column()
These are the settings that control how a column behaves in the database.
primary_key=True
Makes this column the primary key — a unique identifier for every row. The database auto-generates the value (1, 2, 3, …). You never set this yourself.
id: Mapped[int] = mapped_column(primary_key=True)
nullable=True / False
Controls whether the column can be empty (NULL). Default: False (NOT NULL). Using Mapped[Optional[str]] automatically sets nullable=True.
age: Mapped[Optional[int]] = mapped_column()
unique=True
No two rows can have the same value in this column. The database will reject an INSERT that tries to create a duplicate.
email: Mapped[str] = mapped_column(unique=True)
default=value
A Python-side default — if you do not provide a value when creating an object, SQLAlchemy uses this default. Use server_default for a database-side default.
status: Mapped[str] = mapped_column(default="pending")
index=True
Creates a database index on this column. Makes filtering and searching this column much faster, at the cost of slightly slower writes.
city: Mapped[str] = mapped_column(index=True)
name="column_name"
Override the column name in the database. Useful when the Python attribute name must be different from the SQL column name.
user_name: Mapped[str] = mapped_column(name="username")
Real-World Pattern — UUID Primary Keys
Why use UUIDs instead of integers?

Integer primary keys (1, 2, 3…) are simple but expose data. Anyone who sees /users/1 in a URL can guess /users/2 exists, and enumerate every row. UUIDs are globally unique random IDs — they look like a3f8c2d1-… and are safe to expose in URLs and APIs.

UUIDs are also safer in distributed systems — two servers can each generate a UUID and they will never collide, even without coordinating with each other.

Production standard

Most production APIs (Stripe, GitHub, etc.) use UUID or similar opaque IDs. Real-world FastAPI + SQLAlchemy projects use UUID PKs for every model.

import uuid
from uuid import UUID
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column

class Base(DeclarativeBase): pass

# ── Recommended: Python generates the UUID ──
class User(Base):
    __tablename__ = "users"
    id: Mapped[UUID] = mapped_column(primary_key=True, default=uuid.uuid4)
    # default=uuid.uuid4 → Python calls uuid.uuid4() before every INSERT
    # The UUID is set on the object in Python BEFORE the row hits the DB
    name: Mapped[str]

# ── Usage is identical to integer PK ──
user = User(name="Alice")
print(user.id)  # UUID('a3f8c2d1-4b5e-...') — already set before any DB call
session.add(user)
session.commit()

# ── Foreign key pointing to a UUID PK ──
class Order(Base):
    __tablename__ = "orders"
    id:      Mapped[UUID] = mapped_column(primary_key=True, default=uuid.uuid4)
    user_id: Mapped[UUID] = mapped_column(ForeignKey("users.id", ondelete="CASCADE"))
    # ondelete="CASCADE" → deleting the user auto-deletes their orders
Real-World Pattern — Enum Columns
Storing fixed choices safely in the database

Some columns should only ever hold a fixed set of values — a user's role (admin, user), a status (active, suspended), or a media type (image, video). Storing them as raw strings is fragile — a typo like "Admin" silently breaks things.

Enum columns let you define the valid values in Python, and SQLAlchemy enforces them at the Python and database level. Your editor also gives you autocomplete.

Three enum styles — same goal, different storage

ENUM COLUMNS — ALL THREE STYLES
from enum import StrEnum, IntEnum
from sqlalchemy import Enum as SAEnum, SmallInteger
from sqlalchemy.orm import Mapped, mapped_column

# ── Style 1: StrEnum — stored as readable strings ──
class UserStatus(StrEnum):
    ACTIVE    = "active"
    SUSPENDED = "suspended"
    DELETED   = "deleted"

class User(Base):
    __tablename__ = "users"
    id:     Mapped[int]        = mapped_column(primary_key=True)
    status: Mapped[UserStatus] = mapped_column(SAEnum(UserStatus), default=UserStatus.ACTIVE)
    # DB stores: "active" | "suspended" | "deleted" — rejects anything else


# ── Style 2: IntEnum — stored as compact integers ──
class RoleType(IntEnum):
    USER  = 1
    ADMIN = 2
    MOD   = 3

class UserWithRole(Base):
    __tablename__ = "users_role"
    id:   Mapped[int]      = mapped_column(primary_key=True)
    role: Mapped[RoleType] = mapped_column(SmallInteger, default=RoleType.USER)
    # DB stores: 1 | 2 | 3   (SmallInteger is more compact than a string column)

# ── Filtering by enum value ──
from sqlalchemy import select
admins = session.scalars(select(UserWithRole).where(UserWithRole.role == RoleType.ADMIN))
active = session.scalars(select(User).where(User.status == UserStatus.ACTIVE))
Important Distinction — default vs server_default
Where does the default value come from?
DEFAULT= (Python side)
SQLAlchemy calls your Python function/value before the INSERT. The value is set on the object immediately.
created_at = mapped_column(
  default=datetime.utcnow
)

id = mapped_column(
  default=uuid.uuid4
)
✓Value visible in Python before flush
✓Works with any Python type or callable
✗Skipped if you INSERT via raw SQL
vs
SERVER_DEFAULT= (Database side)
The database fills in the default when it executes the INSERT. The column is omitted from the SQL — the DB fills it in.
created_at = mapped_column(
  server_default=func.now()
)

is_active = mapped_column(
  server_default="true"
)
✓Enforced even with raw SQL inserts
✓Can use DB functions like NOW()
✗Value not on Python object until refresh
Best practice
Use both together for timestamp fields: default=datetime.utcnow, server_default=func.now(). Python has the value immediately for logic, AND the database enforces it independently for safety. Real production code always does this.

Relationships — Connecting Models Together

A relationship lets you navigate from one object to related objects directly in Python — without writing a JOIN query. For example, if you have a User object, you can do user.orders to get a list of all their orders. SQLAlchemy handles the JOIN behind the scenes.

Relationships are defined using the relationship() function. They are declared on both sides of the connection using back_populates — this keeps both sides in sync automatically. The foreign key in the database is what makes the connection real; the relationship is just a Python-level convenience on top of it.

3.1 — One-to-Many: using the relationship in code

Once the relationship is defined, you can navigate between objects naturally. user.orders returns a Python list of all Order objects that belong to that user. order.user gives you back the User object that placed the order.

SQLAlchemy automatically generates a JOIN query behind the scenes when you access user.orders. You do not write any SQL. You just access an attribute like a normal Python object.

back_populates keeps both sides in sync

When you append an order to user.orders, SQLAlchemy automatically sets order.user to that user as well. The two sides of the relationship are always consistent with each other — you never have to update both manually.

RELATIONSHIP IN ACTION
with Session(engine) as session:

    # Get a User by primary key
    alice = session.get(User, 1)
    print(alice.name)          # → "Alice"

    # Access all her orders — SQLAlchemy does the JOIN
    print(alice.orders)
    # → [<Order id=1 product='Laptop'>,
    #    <Order id=2 product='Mouse'>]

    # Navigate from order back to user
    order = session.get(Order, 1)
    print(order.user.name)     # → "Alice"
    print(order.product)       # → "Laptop"

    # Add a new order directly via relationship
    new_order = Order(
        product="Webcam", amount=60.00
    )
    alice.orders.append(new_order)
    # SQLAlchemy auto-sets new_order.user_id = alice.id
    session.commit()  # saves the new order ✓
3.2 — Many-to-Many: using an association table

A many-to-many relationship means one record can relate to many of another, and vice versa. For example: a Student can enrol in many Courses, and a Course can have many Students.

The database cannot represent this directly — you need a middle (association) table that stores pairs of IDs. In SQLAlchemy, you define that table separately and pass it as secondary= in relationship(). After that, the relationship works exactly like a one-to-many — you use .append() and access the list normally.

When to use association tables vs association objects

A simple Table (as shown here) is enough when the middle table only has the two foreign keys. If you need extra columns on the join (like enrolment_date), create it as a full model class instead — called an association object.

MANY-TO-MANY — association table pattern
from sqlalchemy import Table, Column, Integer, ForeignKey

# The middle table — just two foreign key columns, nothing else
student_course = Table(
    "student_course",
    Base.metadata,
    Column("student_id", Integer, ForeignKey("students.id")),
    Column("course_id",  Integer, ForeignKey("courses.id")),
)

class Student(Base):
    __tablename__ = "students"
    id:   Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(100))
    courses: Mapped[List["Course"]] = relationship(
        secondary=student_course,
        back_populates="students"
    )

class Course(Base):
    __tablename__ = "courses"
    id:    Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str] = mapped_column(String(200))
    students: Mapped[List["Student"]] = relationship(
        secondary=student_course,
        back_populates="courses"
    )

# Using it — SQLAlchemy handles the join table automatically
with Session(engine) as session:
    student = session.get(Student, 1)
    course  = session.get(Course,  1)
    student.courses.append(course)   # inserts into student_course
    session.commit()

The Session — Your Gateway to the Database

The Session is the ORM's equivalent of Core's Connection. All database operations in the ORM go through a Session. Think of it as a workspace — you add objects to it, modify them, delete them, and then call commit() to write all changes to the database at once.

The Session also acts as an identity map — if you load the same User twice (by id=1), the Session returns the exact same Python object both times, not two separate copies. This ensures your code always works with consistent data.

Session Lifecycle — Step by Step
1
Open Session
Get a Session from the session factory. This grabs a connection from the pool.
Session(engine)
2
Make Changes
Add, modify, or delete objects. Changes are tracked in memory — not yet in the database.
session.add(obj)
3
flush (auto)
Before a query runs, the Session flushes pending changes to the DB inside the transaction so queries see fresh data.
auto-flush
4
commit()
All flushed changes are permanently saved. Transaction is closed. Session is ready for the next operation.
session.commit()
✓
Close Session
Session returns the connection to the pool. Use with Session(engine) as session: to close automatically.
auto on exit
Creating and using a Session

There are two ways to create a Session. Use with Session(engine) as session: for quick scripts — it closes automatically when the block exits. Use sessionmaker() in larger apps (FastAPI, Flask) to create a session factory — a reusable factory that you call each request to get a fresh Session.

Always use sessions as context managers (the with block). This ensures the session is closed and the connection is returned to the pool when you are done, even if an error occurs.

One session per request

In a web application, create one Session per HTTP request, use it for all database operations in that request, and close it when the response is sent. Never share a Session between requests — they are not thread-safe.

TWO WAYS TO CREATE A SESSION
from sqlalchemy import create_engine
from sqlalchemy.orm import Session, sessionmaker

engine = create_engine("sqlite:///shop.db")

# ── WAY 1: Direct Session (scripts, one-offs) ──
with Session(engine) as session:
    # do things with session...
    session.commit()
# session is auto-closed here ✓


# ── WAY 2: sessionmaker (recommended for apps) ──
SessionFactory = sessionmaker(bind=engine)

# In your route / request handler:
with SessionFactory() as session:
    # do things with session...
    session.commit()


# ── ASYNC version (FastAPI, asyncio) ──
from sqlalchemy.ext.asyncio import (
    create_async_engine, async_sessionmaker
)
async_engine = create_async_engine("postgresql+asyncpg://...")
AsyncSession = async_sessionmaker(async_engine)

async with AsyncSession() as session:
    async with session.begin():
        # async database operations...
        pass

CREATE — Adding New Rows to the Database

CREATE

Creating a new row is the most natural ORM operation. You create a Python object of the model class (like creating any Python object), then hand it to the Session with session.add(), and finally call session.commit() to permanently save it to the database.

There is no INSERT statement for you to write. SQLAlchemy looks at the object, sees which table it belongs to, and generates the correct INSERT SQL automatically. After commit(), the object's id attribute is automatically populated with the value the database assigned.

    # Step 1: Create a Python object
    new_user = User(

What session.add() actually does

When you call session.add(obj), the object enters the "pending" state — it is tracked by the session but not yet in the database. When commit() is called, the session flushes all pending objects (generates INSERT SQL) and then commits the transaction. This batching is what makes ORM operations efficient.

READ

Reading data is the most common database operation. The ORM gives you two main tools: session.get() for fetching a single row by its primary key, and session.execute(select(Model)) for writing queries that return multiple rows.

The select(Model) function builds a SELECT statement in Python. You chain methods like .where(), .order_by(), .limit() onto it to add conditions. Then you pass it to session.execute() and call .scalars().all() to get a list of model objects back.

What do .scalars() and .all() do?

session.execute() returns a raw result object. .scalars() tells SQLAlchemy "I want the first column of each row as a Python object" — which gives you your model objects (User, Order, etc.). .all() then collects all results into a Python list. If you want just the first one, use .first() instead of .all().

6.3 — .where() — filter rows by conditions
WHERE — single condition
from sqlalchemy import select

with Session(engine) as session:

    # Users older than 28
    stmt = select(User).where(User.age > 28)
    result = session.execute(stmt).scalars().all()
    for u in result:
        print(u.name, u.age)
    # → Bob 34, Diana 31


    # Find by exact value
    stmt = select(User).where(
        User.city == "London"
    )
    london_users = session.execute(stmt).scalars().all()
    # → [Bob]


    # Find first match only (returns object or None)
    user = session.execute(
        select(User).where(User.name == "Charlie")
    ).scalars().first()
    print(user.city)  # → "Paris"
WHERE — multiple conditions with and_ / or_
from sqlalchemy import select, and_, or_

with Session(engine) as session:

    # AND — both conditions must be true
    stmt = select(User).where(
        and_(
            User.age > 25,
            User.age < 35
        )
    )
    # → Alice(28), Bob(34), Diana(31), Eve(27)


    # OR — at least one condition must be true
    stmt = select(User).where(
        or_(
            User.city == "London",
            User.city == "Paris"

    stmt = select(User).where(

    stmt = select(User).where(
        User.email.like("%@example.com")

ORDERING AND PAGINATION
with Session(engine) as session:

    # Sort by age ascending (youngest first)
    stmt = select(User).order_by(User.age.asc())
    # → Charlie(22), Frank(25), Eve(27), Alice(28), Diana(31), Bob(34)

    stmt = select(User).order_by(User.age.desc())
    # → Bob(34), Diana(31), Alice(28), Eve(27), Frank(25), Charlie(22)

    # Sort by two columns: city first, then age
    stmt = select(User).order_by(User.city, User.age.desc())

    # Limit: only return the first 3 rows
    stmt = select(User).order_by(User.age).limit(3)
    # → Charlie(22), Frank(25), Eve(27)

    # Pagination: OFFSET skips rows — page 2 (rows 4–6)
    stmt = select(User).order_by(User.age).offset(3).limit(3)
    # → Alice(28), Diana(31), Bob(34)

    # Combine: where + order + limit
    stmt = (
        select(User)
        .where(User.age >= 25)
        .order_by(User.age)
        .limit(3)
    )
    result = session.execute(stmt).scalars().all()
    # → Frank(25), Eve(27), Alice(28)

UPDATE — Modifying Existing Rows

UPDATE

Updating a record in the ORM is extremely simple. You load the object from the database, change its attributes directly (like changing a normal Python object's properties), and then call session.commit(). The Session detects the change automatically and generates the UPDATE SQL.

You do not need to tell the Session "hey this object was modified." It tracks all loaded objects and knows exactly which attributes changed — this is called dirty tracking.

    # Step 1: Load the object

UPDATE — bulk update with execute()
from sqlalchemy import update

with Session(engine) as session:

    # Update ALL users from "Berlin" to "Hamburg"
    # without loading them into Python objects first
    # (much faster for large datasets)
    stmt = (
        update(User)
        .where(User.city == "Berlin")
        .values(city="Hamburg")
    )
    result = session.execute(stmt)
    session.commit()
    print(result.rowcount)  # how many rows changed
    # → 1 (Eve's row)


    # Give a 10% raise to all orders over $100
    from sqlalchemy import update
    stmt = (
        update(Order)
        .where(Order.amount > 100)
        .values(amount=Order.amount * 1.1)
    )
    session.execute(stmt)
    session.commit()
Object-level vs bulk update

The load-change-commit approach is clean but loads every object into Python memory. The update(Model).where().values() approach runs a single SQL UPDATE without loading anything — always use bulk update when changing many rows at once.

7.1 — session.rollback() — undoing changes
ROLLBACK — undo uncommitted changes
with Session(engine) as session:
    bob = session.get(User, 2)
    bob.city = "Wrong City"   # oops, this is a mistake

    # Did NOT commit yet — we can undo this!
    session.rollback()

    # bob.city is back to its original value
    print(bob.city)  # → "Toronto" (original value restored)

# Key rule: rollback() undoes ALL uncommitted changes in the session —
# not just the last one. Once commit() is called, rollback() cannot undo it.

You can also use the bulk delete(Model).where() syntax to delete many rows at once without loading them first — this is much faster for deleting in bulk.

Advanced ORM Querying

Beyond simple filtering, the ORM's select() supports the full power of SQL — JOINs across models, GROUP BY with aggregates, sub-queries, and counting. These all look like method chains in Python and produce exactly the SQL you would write by hand.

Query method reference
.scalars().all()
Executes the query and returns all matching model objects as a Python list.
returns: list[ModelClass]
.scalars().first()
Returns only the first matching object, or None if nothing matched. Good for finding a single item.
returns: ModelClass | None
.scalars().one()
Expects exactly one result. Raises NoResultFound if zero rows returned, MultipleResultsFound if more than one.
raises if 0 or 2+ rows
.scalars().one_or_none()
Returns the object if exactly one row matched, or None if zero rows. Still raises if more than one row matched.
returns: ModelClass | None
JOIN — querying across related models

You can use .join() on a select() to combine data from two models. This is the ORM equivalent of writing JOIN in SQL. SQLAlchemy knows which column to join on because you already declared the ForeignKey in the model definition.

You can JOIN and still return full model objects, or you can select specific columns from multiple models at once. The .join() method is smart — if the relationship is defined, you can just pass the target model class and SQLAlchemy figures out the ON clause.

relationship() vs .join()

Use user.orders (the relationship) when you already have a User object and want its orders. Use .join() when you want to filter, count, or aggregate across the relationship in a single query without loading individual objects first.

ORM JOIN — cross-model queries
from sqlalchemy import select

with Session(engine) as session:

    # Get users who have at least one order
    stmt = (
        select(User)
        .join(Order, Order.user_id == User.id)

    users = session.execute(stmt).scalars().all()

        select(User.name, Order.product, Order.amount)

    stmt = (

GROUP BY and Aggregate Functions

Use func from sqlalchemy to call aggregate functions like COUNT, SUM, AVG, MIN, and MAX. These work exactly as they do in SQL. You use .group_by() to group rows, and .having() to filter groups (the same as SQL's HAVING clause).

When using aggregates, you are no longer selecting full model objects — you are selecting specific columns and computed values. So use .all() (not .scalars().all()) to get tuples of results.

GROUP BY + AGGREGATES
from sqlalchemy import select, func

with Session(engine) as session:

    # Count orders per user
    stmt = (
        select(User.name, func.count(Order.id).label("order_count"))
        .join(Order, Order.user_id == User.id)
        .group_by(User.id)
    )
    rows = session.execute(stmt).all()
    for name, count in rows:
        print(f"{name} has {count} orders")


    # Total spend per user — only users who spent > $100
    stmt = (
        select(User.name, func.sum(Order.amount).label("total"))
        .join(Order, Order.user_id == User.id)
        .group_by(User.id)
        .having(func.sum(Order.amount) > 100)
        .order_by(func.sum(Order.amount).desc())
    )
    rows = session.execute(stmt).all()
    # → Alice $1029.98, Bob $79.99 (filtered out)


    # Average order amount across all users
    avg = session.execute(
        select(func.avg(Order.amount))
    ).scalar()   # .scalar() → single value
    print(f"Avg order: ${avg:.2f}")

from sqlalchemy import select, func, exists

with Session(engine) as session:

    # Count total number of users
    total = session.execute(select(func.count())).scalar()
    total = session.execute(select(func.count(User.id))).scalar()
    print(f"Total users: {total}")   # → 5

    # Check if a user with this email already exists
    email = "alice@example.com"
    already_exists = session.execute(
        select(exists().where(User.email == email))
    ).scalar()
    print(already_exists)   # → True

    # Find users who have at least one shipped order
    subq = (
        select(Order.user_id)
        .where(Order.status == "shipped")
        .scalar_subquery()
    )
    stmt = select(User).where(User.id.in_(subq))
    users = session.execute(stmt).scalars().all()
    for u in users:
        print(u.name)  # → only users with shipped orders

    # Return the user with the most orders (combined query)
    stmt = (
        select(User.name, func.count(Order.id).label("n"))
        .join(Order, Order.user_id == User.id)
        .group_by(User.id)
        .order_by(func.count(Order.id).desc())
        .limit(1)
    )
    name, count = session.execute(stmt).one()
    print(f"{name} has the most orders: {count}")
Eager Loading — avoid the N+1 query problem

The N+1 problem is a common performance trap. Imagine you load 100 users, then loop over them and access user.orders for each. That is 1 query to get the users, then 100 more queries to get their orders — 101 queries total. This is extremely slow.

The fix is eager loading — tell SQLAlchemy to load the related objects in the same query using selectinload(). This runs 2 queries total (one for users, one for all their orders combined) regardless of how many users there are.

selectinload vs joinedload

selectinload runs a second SELECT ... WHERE id IN (...) — the recommended default. joinedload uses a single SQL JOIN — better when you need to filter on the related model's columns in the same query.

EAGER LOADING — avoid N+1
from sqlalchemy.orm import selectinload

with Session(engine) as session:

    # ❌ BAD — N+1 queries (1 + one per user)
    users = session.execute(select(User)).scalars().all()
    for u in users:
        print(u.orders)  # ← fires a new query each time!


    # ✅ GOOD — 2 queries total, no matter how many users
    stmt = (
        select(User)
        .options(selectinload(User.orders))
    )
    users = session.execute(stmt).scalars().all()
    for u in users:
        print(u.orders)  # ← no extra query, already loaded


    # joinedload — one query with LEFT OUTER JOIN
    from sqlalchemy.orm import joinedload
    stmt = (
        select(User)
        .options(joinedload(User.orders))
        .where(User.city == "New York")
    )
    users = session.execute(stmt).unique().scalars().all()
Putting it all together — full working example
COMPLETE EXAMPLE — define models, create tables, run all CRUD
from __future__ import annotations
from typing import List, Optional
from sqlalchemy import create_engine, String, Float, ForeignKey, select, func
from sqlalchemy.orm import (
    DeclarativeBase, Mapped, mapped_column, relationship, Session
)

# 1. Define Base and models
class Base(DeclarativeBase): pass

class User(Base):
    __tablename__ = "users"
    id:    Mapped[int]           = mapped_column(primary_key=True)
    name:  Mapped[str]           = mapped_column(String(100))
    email: Mapped[str]           = mapped_column(String(200), unique=True)
    age:   Mapped[Optional[int]]
    city:  Mapped[Optional[str]]  = mapped_column(String(100))
    orders:Mapped[List["Order"]] = relationship(back_populates="user")

class Order(Base):
    __tablename__ = "orders"
    id:      Mapped[int]   = mapped_column(primary_key=True)
    user_id: Mapped[int]   = mapped_column(ForeignKey("users.id"))
    product: Mapped[str]   = mapped_column(String(200))
    amount:  Mapped[float]
    status:  Mapped[str]   = mapped_column(String(50), default="pending")
    user:    Mapped["User"] = relationship(back_populates="orders")

# 2. Create engine and tables
engine = create_engine("sqlite:///shop.db", echo=True)
Base.metadata.create_all(engine)   # creates users + orders tables

# 3. INSERT rows
with Session(engine) as session:
    alice = User(name="Alice", email="alice@ex.com", age=28, city="New York")
    bob   = User(name="Bob",   email="bob@ex.com",   age=34, city="London")
    session.add_all([alice, bob])
    session.commit()
    alice.orders.append(Order(product="Laptop", amount=999.99, status="shipped"))
    session.commit()

# 4. READ and QUERY
with Session(engine) as session:
    alice = session.get(User, 1)
    print(alice.name, alice.orders[0].product)  # Alice  Laptop
    young = session.execute(select(User).where(User.age < 30)).scalars().all()
    print([u.name for u in young])  # → ['Alice']

# 5. UPDATE
with Session(engine) as session:
    user = session.get(User, 2)
    user.city = "Manchester"
    session.commit()

# 6. DELETE
with Session(engine) as session:
    user = session.get(User, 2)
    session.delete(user)
    session.commit()
    print("Bob deleted")

# 7. AGGREGATE
with Session(engine) as session:
    total_users  = session.execute(select(func.count(User.id))).scalar()
    total_orders = session.execute(select(func.count(Order.id))).scalar()
    avg_amount   = session.execute(select(func.avg(Order.amount))).scalar()
    print(f"Users: {total_users}, Orders: {total_orders}, Avg: ${avg_amount:.2f}")
