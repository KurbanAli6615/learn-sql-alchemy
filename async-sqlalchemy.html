AsyncSQLAlchemy

Modern Python apps handle thousands of requests at once using asyncio. Async SQLAlchemy lets your database queries participate in that model — so your app never blocks waiting for the database. This module teaches the full async stack from engine to session to query.

Why Async? The Problem Sync SQLAlchemy Has
Understanding what "blocking" means — and why it matters under load.
01 / 04

Sync Blocks. Async Waits Efficiently.

A regular (synchronous) database call blocks. When your code says session.execute(query), the entire Python thread stops and waits — doing nothing — until the database replies. On a slow query that takes 200ms, your server can't handle any other request on that thread during those 200ms.

Async works differently. When you await session.execute(query), Python says "okay, go fetch that — I'll do other work while you're gone." The server can handle dozens of other requests while waiting for the database reply.

For apps that serve many concurrent users, this is the difference between needing 100 server threads and needing 4.

The restaurant analogy

Sync is a waiter who takes one order, stands at the kitchen window waiting for the food, then brings it out — then takes the next order. Async is a waiter who takes 10 orders, submits them all, and delivers each plate as it comes out. Same kitchen — 10× the throughput.

Setting Up the Async Engine and Session
Installation, engine configuration, session factory, and the critical expire_on_commit=False setting.
02 / 04

The Three Pieces: Engine, Session Factory, Session

Setting up async SQLAlchemy requires three things, each built on top of the previous one. The engine manages the connection pool to the database. The session factory is a reusable template for creating sessions. The session is the object your code uses for every query.

You create the engine and session factory once at startup, then create a new session for each request (or unit of work). Sessions are cheap and short-lived — engines are expensive and long-lived.

Install the async driver first

Async SQLAlchemy requires an async database driver. For PostgreSQL use asyncpg. For MySQL use aiomysql. For SQLite use aiosqlite. Without this, create_async_engine will fail.

expire_on_commit=False is essential

By default, SQLAlchemy expires all object attributes after a commit — it assumes you'll immediately reload them in the same sync context. In async, attributes are loaded lazily, so accessing an expired attribute outside the session causes a MissingGreenlet error. Setting expire_on_commit=False disables this and keeps values accessible after commit.

INSTALLATION + FULL SETUP — database.py
# pip install sqlalchemy asyncpg

from sqlalchemy.ext.asyncio import (
    create_async_engine,   # async engine
    async_sessionmaker,    # async session factory
    AsyncSession,          # async session type hint
)
from sqlalchemy.orm import DeclarativeBase

# ── Step 1: Database URL — note +asyncpg ──
DATABASE_URL = (
    "postgresql+asyncpg://"
    "user:password@localhost:5432/mydb"
)

# ── Step 2: Create the async engine (once, at startup) ──
engine = create_async_engine(
    DATABASE_URL,
    pool_pre_ping=True,   # test connections before use
    pool_recycle=3600,    # refresh connections hourly
    pool_size=10,          # max permanent connections
    max_overflow=20,       # extra burst connections
    echo=False,            # set True to log all SQL
)

# ── Step 3: Session factory (once, at startup) ──
AsyncSessionLocal = async_sessionmaker(
    engine,
    expire_on_commit=False,  # CRITICAL for async — see callout
    class_=AsyncSession,
)

# ── Step 4: Declarative base for your models ──
class Base(DeclarativeBase): pass

The Async Session Lifecycle — Two Patterns
Both patterns are correct. The begin() pattern auto-commits on success and auto-rolls back on exception — use it for most cases. The manual pattern gives you more control.
PATTERN 1 — auto-commit with begin()
# session.begin() auto-commits on exit,
# auto-rollbacks if an exception is raised

async def create_user(name: str):
    async with AsyncSessionLocal() as session:
        async with session.begin():   # auto-commit here
            user = User(name=name)
            session.add(user)
            # commit happens automatically on __aexit__
            # rollback happens automatically on exception
        return user  # safe because expire_on_commit=False


# Shorthand — combine both context managers
async def get_user(user_id: int):
    async with AsyncSessionLocal() as session, \
               session.begin():
        return await session.get(User, user_id)
PATTERN 2 — manual commit / rollback
# Full manual control — useful when you need
# to inspect results before deciding to commit

async def transfer_funds(from_id, to_id, amount):
    async with AsyncSessionLocal() as session:
        try:
            sender = await session.get(Account, from_id)
            receiver = await session.get(Account, to_id)

            sender.balance -= amount
            receiver.balance += amount

            await session.commit()   # explicit commit
        except Exception:
            await session.rollback() # explicit rollback
            raise

Async Queries — All the Methods You Need
How to write SELECT, INSERT, UPDATE, and DELETE in async. The same select() API — just with await.

Async CRUD — Every Query Type With Code

The query-building API (select(), update(), delete()) is identical in async. The only difference is that you await the session.execute() call that sends it to the database.

The result-fetching methods (.scalar(), .scalars(), .all(), .one()) are called on the result object and are not awaited — they just process data already in memory.

AsyncSession method reference
MethodReturnsUse when
await session.execute(stmt) Result Any query — then call .scalars(), .scalar(), etc.
await session.scalar(stmt) Model | None Get exactly one object or None
await session.scalars(stmt) ScalarResult Get multiple objects — call .all() after
await session.get(Model, pk) Model | None Fetch by primary key — checks identity map first
session.add(obj) None Stage a new object for INSERT — not awaited
await session.flush() None Send pending SQL to DB without committing
await session.commit() None Commit the current transaction
await session.rollback() None Undo everything since last commit
await session.refresh(obj) None Reload an object's attributes from the DB
await session.delete(obj) None Mark an object for DELETE on next flush
FULL ASYNC CRUD EXAMPLES
from sqlalchemy import select, update, delete

# ── CREATE ──
async def create_user(session: AsyncSession, name: str):
    user = User(name=name, email=f"{name.lower()}@example.com")
    session.add(user)           # not awaited — just stages the object
    await session.flush()        # sends INSERT, gets auto-generated id
    return user


# ── READ ONE ──
async def get_user(session: AsyncSession, user_id: int):
    return await session.get(User, user_id)  # uses identity map cache


# ── READ MANY WITH FILTER ──
async def get_active_users(session: AsyncSession):
    result = await session.execute(
        select(User)
        .where(User.is_active.is_(True))
        .order_by(User.created_at.desc())
    )
    return result.scalars().all()    # .scalars().all() not awaited


# ── READ SINGLE VALUE ──
async def find_by_email(session: AsyncSession, email: str):
    return await session.scalar(
        select(User).where(User.email == email)
    )  # returns User or None directly


# ── UPDATE ──
async def deactivate_user(session: AsyncSession, user_id: int):
    user = await session.get(User, user_id)
    if user:
        user.is_active = False   # ORM tracks this change
        await session.flush()     # sends UPDATE


# ── BULK UPDATE ──
async def expire_old_sessions(session: AsyncSession, before):
    await session.execute(
        update(UserSession)
        .where(UserSession.expires_at <= before)
        .values(is_expired=True)
    )


# ── DELETE ──
async def delete_user(session: AsyncSession, user_id: int):
    user = await session.get(User, user_id)
    if user:
        await session.delete(user)
        await session.flush()

Relationships in Async — The Biggest Gotcha
Lazy loading silently breaks in async. Learn what goes wrong and exactly how to fix it.
04 / 04

Lazy Loading Does Not Work in Async — Here's Why

In sync SQLAlchemy, accessing user.orders on a loaded user automatically fires a new SQL query to fetch the orders. This is lazy loading — it happens silently, behind the scenes.

In async SQLAlchemy, this silent database call is not allowed. Async requires all I/O to be explicitly awaited. A lazy load triggered by a simple attribute access has no await, so SQLAlchemy cannot run the query — and raises a MissingGreenlet error instead.

The fix is simple: always load relationships eagerly in async. Use selectinload() or joinedload() in your query so the related data is fetched before you access it.

MissingGreenlet — the async lazy load error

If you see sqlalchemy.exc.MissingGreenlet or greenlet_spawn has not been called, it means you accessed a lazy-loaded relationship outside an async context. Fix: add selectinload() or joinedload() to your query.

LAZY LOADING — BREAKS IN ASYNC
# ❌ This FAILS in async — lazy load triggered outside await
async def get_user_with_orders_broken(session, user_id):
    user = await session.get(User, user_id)
    # user.orders triggers a new SQL query
    # but there's no 'await' — CRASH!
    return user.orders  # → MissingGreenlet error


# ✅ Fix 1: selectinload — separate SELECT for each relationship
from sqlalchemy.orm import selectinload

async def get_user_with_orders(session, user_id):
    result = await session.execute(
        select(User)
        .options(selectinload(User.orders))  # load orders upfront
        .where(User.id == user_id)
    )
    user = result.scalar_one_or_none()
    return user.orders  # ✅ already loaded — no lazy call needed


# ✅ Fix 2: joinedload — single SQL with LEFT JOIN
from sqlalchemy.orm import joinedload

async def get_user_joined(session, user_id):
    result = await session.execute(
        select(User)
        .options(joinedload(User.profile))  # good for one-to-one
        .where(User.id == user_id)
    )
    return result.unique().scalar_one_or_none()
Which Loading Strategy to Use in Async
All three strategies work in async — the rule is just that you must choose one explicitly. Never rely on lazy loading.
Strategy How it works Best for Async safe?
selectinload() 1 extra SELECT per relationship. Uses IN clause. Collections (one-to-many, many-to-many) ✅ Yes
joinedload() Adds a LEFT OUTER JOIN to the main query. Single related objects (many-to-one, one-to-one) ✅ Yes
lazy="select" Fires a SELECT when the attribute is first accessed. Sync code only ❌ MissingGreenlet
lazy="raise" Raises an error on access — forces explicit loading. Catching accidental lazy loads in async ⚠️ Raises on access
NESTED EAGER LOADING IN ASYNC
from sqlalchemy.orm import selectinload, joinedload, load_only

# Load user → orders → each order's product
result = await session.execute(
    select(User)
    .options(
        selectinload(User.orders)           # load orders
        .selectinload(Order.products),       # then each order's products
        joinedload(User.profile)              # and the profile via JOIN
    )
    .where(User.is_active.is_(True))
)
users = result.unique().scalars().all()
# .unique() needed when joinedload is used — deduplicates rows

# Load only specific columns (performance) 
result = await session.execute(
    select(User)
    .options(
        load_only(User.id, User.name, User.email),
        selectinload(User.orders)
        .load_only(Order.id, Order.total)
    )
)
Use lazy="raise" in models

Set lazy="raise" on relationships in your model definitions. This makes SQLAlchemy immediately raise an error if you accidentally trigger a lazy load — instead of silently running a query in sync code or crashing with a cryptic error in async. It forces you to be explicit.

ASYNC-SAFE MODEL DEFINITION
class User(Base):
    __tablename__ = "users"
    id:   Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str]

    # lazy="raise" — forces you to always use
    # selectinload() or joinedload() in queries
    orders: Mapped[List["Order"]] = relationship(
        back_populates="user",
        lazy="raise",  # crash loudly if lazy-loaded
    )
Common Async SQLAlchemy Mistakes — and How to Fix Them
These are the four errors every developer hits when first switching from sync to async SQLAlchemy.
💥
MissingGreenlet on attribute access
Cause: Accessing a lazy-loaded relationship outside an async context.

Fix: Add selectinload() or joinedload() to your query before the session closes.
🚫
Attributes are None after commit
Cause: expire_on_commit=True (the default) expires all attributes after commit.

Fix: Set expire_on_commit=False in async_sessionmaker(). Always do this in async apps.
🔗
Using sync engine with async session
Cause: Mixing create_engine() with AsyncSession.

Fix: Always pair create_async_engine() with async_sessionmaker(). Also make sure the DB URL uses +asyncpg.
⏳
Forgetting await on session methods
Cause: Writing session.execute() instead of await session.execute(). The call returns a coroutine object, not a result.

Fix: Any session method that hits the DB needs await. Only session.add() and .expire_all() do not.
🔄
Sharing a session across tasks
Cause: One session shared between multiple asyncio.Tasks or coroutines running concurrently.

Fix: Each request or task gets its own session. Sessions are not thread-safe or task-safe. Create and close them per unit of work.
📦
Using scoped_session in async
Cause: scoped_session relies on thread-local storage — useless in async where many tasks share one thread.

Fix: Use async_sessionmaker and pass the session as a parameter or via dependency injection (FastAPI's Depends).
