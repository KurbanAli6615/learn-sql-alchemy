Eager Loading vs. Lazy Loading
Why the default behaviour causes the N+1 problem, visualised as a timeline, and how the three eager strategies each solve it.

Lazy Loading — The Default That Hides a Problem

By default, SQLAlchemy uses lazy loading — related objects are not fetched until you access them in Python. The moment you write user.addresses inside a loop, SQLAlchemy fires a brand-new SELECT for each iteration.

This creates the N+1 problem: one query to load N users, then one more query per user when you touch their addresses = N+1 total queries. The solution is eager loading — you tell SQLAlchemy to fetch the related data alongside the main query so no extra queries fire in the loop.

The N+1 problem in one sentence

1 query to load N records + N queries to load their relationships = N+1 database calls. With 1,000 users this becomes 1,001 calls where 2 would suffice.

LAZY vs EAGER — THE DIFFERENCE
# ❌ LAZY — N+1 queries
users = session.query(User).all()
for user in users:
    print(user.addresses)   # fires a SELECT each time


# ✅ EAGER — 1 query with a JOIN
from sqlalchemy.orm import joinedload

users = session.query(User).options(
    joinedload(User.addresses)
).all()
for user in users:
    print(user.addresses)   # already loaded, no query


# ✅ EAGER — 2 queries with selectinload (recommended)
from sqlalchemy.orm import selectinload

users = session.query(User).options(
    selectinload(User.addresses)
).all()
for user in users:
    print(user.addresses)   # already loaded, no query

Three Eager Strategies — Choose by Use Case
Each solves N+1 differently. Your choice depends on collection size and whether you need to filter by the related table.
Joined joinedload()
Performs a LEFT OUTER JOIN — loads parents and children in a single query. Best when you need to filter by the child's columns.
session.query(User).options(
  joinedload(User.addresses)
).all()
1 query total ✓
Subquery subqueryload()
Second query using WHERE id IN (subquery). Better for large collections — avoids the row duplication that joinedload causes.
session.query(User).options(
  subqueryload(User.addresses)
).all()
2 queries total ✓
Selectin selectinload()
SELECT WHERE id IN (…) — the recommended default for SQLAlchemy 2.0. Simpler SQL than subqueryload, no duplication.
session.query(User).options(
  selectinload(User.addresses)
).all()
2 queries total ✓

Using Joins Efficiently
How to let the database combine tables at query time instead of doing it in Python with multiple round-trips.

Efficient Joins — One Trip Instead of Many

Joins combine rows from two or more tables based on a related column. The database executes the combination using its indexes — far faster than fetching both tables separately and merging them in Python.

The most common mistake is running two sequential queries then combining results with Python list comprehensions. A single .join() call consolidates both into one network round-trip and lets the database optimise the execution plan.

The database is better at joins than Python is

The DB uses B-tree indexes, hash joins, and query planner optimisations to combine tables. Python iterates over two lists in memory. At scale, the DB wins by orders of magnitude.

JOIN — QUERY AND FILTER PATTERNS
# Inner join — only rows that match on both sides
results = session.query(User, Address).\
    join(Address).\
    filter(User.name == 'Alice').\
    all()

for user, address in results:
    print(user.name, address.city)


# Left outer join — include users with no address
results = session.query(User, Address).\
    outerjoin(Address).\
    all()


# Explicit join condition
results = session.query(User, Address).\
    join(Address, User.id == Address.user_id).\
    all()


# Chain multiple joins
results = session.query(User).\
    join(User.orders).\
    join(Order.items).\
    filter(Item.price > 100).\
    all()

Why this matters at scale

With two queries, all user IDs travel across the network before the address lookup starts. With one JOIN, the database filters on both tables simultaneously using their indexes — the result set sent over the wire is already the final answer.

Caching Strategies
Two approaches — dogpile.cache for production-grade query caching and lru_cache for simple in-process memoisation.
03 / 04

Caching — Stop Asking the Database the Same Question Twice

Eager loading and joins reduce the number of queries per request. Caching eliminates database calls entirely for data that hasn't changed. If a user profile looked the same 30 seconds ago, there is no reason to hit the database again.

Two patterns cover most cases: dogpile.cache — production-grade, works with Redis or Memcached, configurable TTL per function — and lru_cache — Python's built-in, in-process only, resets on restart. Start with lru_cache, upgrade to dogpile.cache when you need shared caching across multiple processes.

Best data for caching

Cache data that is read often, written rarely: user profiles, product catalogues, config values, country lists. Do not cache real-time data like inventory count, order status, or live prices.

Pattern 1 — Query Caching with dogpile.cache
Production-grade. Redis or Memcached backend. Configurable TTL per function. Shared across processes.
dogpile.cache
from dogpile.cache import make_region

# Configure once — swap backend without changing code
region = make_region().configure(
    'dogpile.cache.memcached',
    expiration_time=3600,    # 1-hour TTL
    arguments={'url': ["127.0.0.1"]}
)

# Decorate — result cached on first call
@region.cache_on_arguments()
def get_user_by_name(name):
    return session.query(User).\
        filter_by(name=name).first()

# Call 1 — hits DB, stores in Memcached
user = get_user_by_name("Alice")

# Call 2 — returns from cache, no DB
user = get_user_by_name("Alice")

# Different arg = different cache key
user = get_user_by_name("Bob")  # cache miss
Pattern 2 — Result Set Caching with lru_cache
Built into Python. In-process only. Resets on restart. Best for small stable datasets.
lru_cache + pickle
import pickle
from functools import lru_cache

# Cache up to 128 unique results in memory
@lru_cache(maxsize=128)
def get_all_users():
    users = session.query(User).all()
    return pickle.dumps(
        [user.__dict__ for user in users]
    )

def fetch_users():
    return pickle.loads(get_all_users())

# Invalidate when data changes
def create_user(name, email):
    user = User(name=name, email=email)
    session.add(user)
    session.commit()
    get_all_users.cache_clear()  # ← important!

# Inspect cache performance
print(get_all_users.cache_info())
# CacheInfo(hits=12, misses=1, maxsize=128, currsize=1)
lru_cache vs dogpile.cache

Use lru_cache for single-process apps. Use dogpile.cache with Redis when the cache must survive restarts, be shared across workers, or expire at predictable intervals.

Profiling, Bulk Ops, Indexing & Pooling

These four techniques support the first three. Use EXPLAIN ANALYZE to find which queries need optimising, bulk operations to handle mass writes, indexes to speed up WHERE filters, and pool tuning to serve concurrent requests without exhausting connections.

🔍
Query Execution Plans
Use EXPLAIN ANALYZE to see exactly how the DB runs a query — which indexes are used, how many rows are scanned, where the time is spent. Run it before adding an index to confirm it is actually needed.
⚡
Bulk Operations
Updating or inserting many rows in a loop is slow. A single .update() with WHERE id IN (…) sends one SQL statement — bypasses per-object ORM overhead and is orders of magnitude faster.
📑
Proper Indexing
Every column in a WHERE, ORDER BY, or JOIN ON clause should have an index. Add them in __table_args__ so Alembic tracks them as part of the schema.
🔌
Connection Pool Tuning
The default pool is conservative. Tune pool_size, max_overflow, and pool_recycle so the engine handles concurrent web requests without waiting for a free connection.
How to Read an EXPLAIN ANALYZE Output
The output tells you whether the database used an index or did a slow full scan — and what to do about it.

EXPLAIN ANALYZE + BULK UPDATE
from sqlalchemy import text

# ── Run EXPLAIN ANALYZE on any slow query ──
with engine.connect() as conn:
    plan = conn.execute(text(
        "EXPLAIN ANALYZE SELECT * FROM users WHERE name = 'Alice'"
    ))
    for row in plan:
        print(row[0])
    # Look for "Seq Scan" → means you need an index
    # Look for "Index Scan" → already optimised


# ── Bulk update — one statement, not a loop ──
session.query(User).\
    filter(User.id.in_([u.id for u in users])).\
    update(
        {"status": "active"},
        synchronize_session=False
    )
session.commit()
# UPDATE users SET status='active' WHERE id IN (…)
# One query regardless of list size
INDEXES IN MODEL + POOL TUNING
from sqlalchemy import Index, create_engine

# ── Indexes in __table_args__ ──
class User(Base):
    __tablename__ = 'users'
    id    = Column(Integer, primary_key=True)
    name  = Column(String(50))
    email = Column(String(50))

    __table_args__ = (
        Index('ix_user_name',  'name'),
        Index('ix_user_email', 'email'),
    )


# ── Connection pool tuning ──
engine = create_engine(
    'postgresql://user:pass@localhost/dbname',
    pool_size=10,       # persistent connections
    max_overflow=20,    # burst connections under load
    pool_recycle=3600,  # recycle every 1 hour
)
# max 30 simultaneous connections before queuing
Always measure first

Run EXPLAIN ANALYZE before adding an index, enabling eager loading, or adding a cache. Optimization without measurement wastes time and can make things worse.

Pagination — Returning Results in Pages, Not All at Once

Imagine a table with 100,000 products. If your API returns all of them in one response, it will be slow, memory-heavy, and crash slow clients. Pagination splits the result into pages — the client asks for page 1, then page 2, and so on.

There are two common strategies. Offset/limit is simple and works for most use cases. Cursor-based is more efficient for very large tables or real-time feeds where new rows are being added while the user is paginating.

Always paginate list endpoints

Never return an unbounded list from an API. Even if your table has only 500 rows today, it will eventually have 500,000. Pagination is a habit, not a feature.

OFFSET / LIMIT PAGINATION
from sqlalchemy import select, func

# ── Basic offset/limit ──
def get_products(session, page: int = 1, per_page: int = 20):
    offset = (page - 1) * per_page
    items = session.scalars(
        select(Product)
        .order_by(Product.created_at.desc())
        .limit(per_page)
        .offset(offset)
    ).all()
    return items

# ── With total count (for "Page X of Y") ──
def get_products_with_total(session, page: int = 1, per_page: int = 20):
    offset = (page - 1) * per_page

    total = session.scalar(select(func.count()).select_from(Product))
    items = session.scalars(
        select(Product)
        .order_by(Product.id)
        .limit(per_page)
        .offset(offset)
    ).all()

    return {
        "items":      items,
        "total":      total,
        "page":       page,
        "per_page":   per_page,
        "total_pages": -(-total // per_page)  # ceiling division
    }
CURSOR-BASED PAGINATION
# ── Cursor-based: no OFFSET — uses last seen ID ──
# Better for large tables and real-time feeds

def get_products_after(session, after_id=None, limit: int = 20):
    query = select(Product).order_by(Product.id).limit(limit)

    if after_id:
        # Only rows AFTER the last ID the client saw
        query = query.where(Product.id > after_id)

    items = session.scalars(query).all()
    next_cursor = items[-1].id if len(items) == limit else None

    return {"items": items, "next_cursor": next_cursor}
    # Client passes next_cursor as after_id in their next request
    # No OFFSET means no "skipping rows" performance cost
