What is a Relational Database?

Think of a database as a well-organized filing cabinet for all your app's data. A relational database organizes data into tables — and the special part is that these tables can be connected to each other. That's the "relational" in the name.

Each table stores one type of thing. A users table holds all your users. An orders table holds all your orders. When an order needs to know who placed it, it stores the user's ID — creating a link between the two tables.

Table
A grid of rows and columns — like a spreadsheet for one type of data.
Row (Record)
One single entry in a table — one user, one order, one product.
Primary Key
A unique ID for each row. No two rows can share the same primary key.
Foreign Key
A column that points to a primary key in another table — this creates the relationship.
Why This Matters

Instead of copying Alice's name into every order row, you store her user_id = 1. One change to her name updates everywhere automatically. No duplication, no inconsistency.

Why Do We Need an ORM?

Without an ORM, you write raw SQL strings directly in your Python code. It works — but look at the pain:

WITHOUT ORM — MESSY 
# SQL mixed into Python strings
cursor.execute("""
    INSERT INTO users (name, email)
    VALUES (%s, %s)
""", ("Alice", "alice@example.com"))

rows = cursor.fetchall()
# manually map DB rows to Python dicts
users = [{"id": r[0], "name": r[1]} for r in rows]

With SQLAlchemy ORM, you work with Python objects. No SQL strings, no manual row mapping:

WITH SQLALCHEMY ORM — CLEAN 
# Pure Python — SQLAlchemy handles SQL
user = User(name="Alice", email="alice@example.com")
session.add(user)
session.commit()

# Query reads like English
users = session.query(User).filter(
    User.name == "Alice"
).all()
The ORM Advantage

Your Python classes become database tables. Rename a column in the class and it renames everywhere — no hunting through SQL strings. No typos, no manual mapping, no mixed concerns.

03 / 03

Meet SQLAlchemy

SQLAlchemy is Python's most powerful and widely-used database library — created in 2005 and still the gold standard today. It's not just an ORM; it's a complete database toolkit with two layers.

You can choose how much SQL control you want. Use Core when you need raw power. Use the ORM when you want to stay in pure Python. Most real-world apps use both.

SQLAlchemy Core
Low-level SQL toolkit. Build queries as Python expressions. Full control.
SQLAlchemy ORM
High-level layer. Maps Python classes to DB tables. Ideal for apps.
Multi-Database
PostgreSQL, MySQL, SQLite, Oracle, SQL Server — same code, any DB.
Battle-Tested
Used at Reddit, Yelp, Survey Monkey, and thousands of Python projects.
SQLALCHEMY 2.0 — QUICK LOOK 
from sqlalchemy import create_engine, String, Integer
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, Session

class Base(DeclarativeBase): pass

class User(Base):
    __tablename__ = "users"
    id:    Mapped[int] = mapped_column(Integer, primary_key=True)
    name:  Mapped[str] = mapped_column(String(100))
    email: Mapped[str] = mapped_column(String(200))

engine = create_engine("sqlite:///myapp.db")
Base.metadata.create_all(engine)

with Session(engine) as session:
    user = User(name="Alice", email="alice@example.com")
    session.add(user)
    session.commit()
