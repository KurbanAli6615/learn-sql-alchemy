How a Request Flows Through the Stack

Every API request follows the same path. FastAPI receives it, resolves all dependencies (including your database session), calls your route handler, and then cleans up automatically. Understanding this flow tells you exactly where to put each piece of your code.

The key mechanism is dependency injection via Depends(). You declare what your route needs, and FastAPI constructs and provides it. Your route function never has to create or close a session — it just receives one and uses it.

Structuring your app in layers also makes it easy to test. Services can be tested without FastAPI. Routes can be tested without a real database. Each layer has a clear job.

Why layers?

Without layers, routes become long, hard to test, and hard to reuse. A service layer separates "what the API endpoint does" (route) from "how data is fetched/stored" (service). Each route stays under 10 lines. Each service function stays under 20.

Request → Response flow
1
HTTP Request arrives
FastAPI matches the URL to a route. It reads the decorator to discover what dependencies the function needs.
POST /users → create_user()
2
Depends(db_session) resolves
FastAPI calls your db_session() generator. It creates the AsyncSession, begins a transaction, and yields the session into the route.
AsyncSession created + transaction begun
3
Route handler runs
Your route function receives the session. It calls the service layer, which does the actual database work using the session.
await user_service.create(session, data)
4
Response is returned
FastAPI serialises the return value using the response_model Pydantic schema and sends the HTTP response.
201 Created { "id": 1, "name": "..." }
5
Cleanup: session closed automatically
FastAPI resumes the db_session() generator after the route returns. The async with block exits, the session is closed, and the connection is returned to the pool. On exception, rollback happens here.

The db_session Generator — Line by Line

The db_session function is the bridge between FastAPI and SQLAlchemy. It's an async generator — a function that uses yield to pause, hand control to FastAPI, and then resume after the request is done.

FastAPI calls this function before running the route, injects the yielded value as the db parameter, runs the route, and then resumes the generator for cleanup. The try/finally pattern ensures the session always closes — even if an exception occurs.

This is called a dependency with cleanup in FastAPI. The yield is the boundary between setup and teardown.

One session per request — always

Create a fresh session at the start of each request, use it throughout, and close it at the end. Never share sessions between requests or create sessions inside service functions. The session is created once by the dependency and passed down through the call stack.

core/db.py — the complete production db.py
from collections.abc import AsyncIterator
from sqlalchemy.ext.asyncio import (
    AsyncSession, async_sessionmaker, create_async_engine,
)

DATABASE_URL = "postgresql+asyncpg://user:pass@localhost/db"

# Engine — created ONCE at startup, shared forever
engine = create_async_engine(
    DATABASE_URL,
    pool_pre_ping=True,   # test connections before use
    pool_recycle=3600,    # refresh connections every hour
    pool_size=10,
    max_overflow=20,
)

# Session factory — configured ONCE at startup
async_session = async_sessionmaker(
    engine,
    expire_on_commit=False,  # essential for async
)


# ── The FastAPI dependency ──
async def db_session() -> AsyncIterator[AsyncSession]:
    # Everything BEFORE yield runs before the route handler
    async with async_session() as session:
        async with session.begin():
            # yield hands the session to the route function
            yield session
            # After this point: auto-commit on success,
            # auto-rollback on exception (handled by begin())
    # session.close() is automatic via 'async with'


# ── Type alias — cleaner route signatures ──
from typing import Annotated
from fastapi import Depends

DBSession = Annotated[AsyncSession, Depends(db_session)]
# Usage: async def my_route(db: DBSession): ...
Session Lifecycle: Success vs Exception
The session behaves differently depending on whether the route succeeds or raises an exception. Both paths are handled automatically by the context managers.
✅ Successful request
1
Request arrives, db_session() called
2
AsyncSession created, transaction begun
3
Session yielded to route handler
4
Route runs successfully, returns response
5
Transaction committed automatically
6
Session closed, connection returned to pool
❌ Exception in route
1
Request arrives, db_session() called
2
AsyncSession created, transaction begun
3
Session yielded to route handler
4
Route raises an exception (e.g. HTTPException)
5
Transaction rolled back automatically — no partial writes
6
Session closed, connection returned, error response sent

Without a Service Layer vs With One

See why routes with embedded business logic become unmaintainable, and how the service layer pattern fixes it.
❌ Without service layer
• SQL queries live inside route functions
• Business logic scattered across routes
• Impossible to test without HTTP
• Duplicate queries across routes
• Route files grow to 500+ lines
❌ FAT ROUTE — hard to test
# Route doing everything itself — bad
@router.post("/users")
async def create_user(
    data: UserCreate,
    db: DBSession,
):
    # checking duplicates — business logic in route!
    existing = await db.scalar(
        select(User).where(User.email == data.email)
    )
    if existing:
        raise HTTPException(400, "Email taken")

    # hashing — also business logic in route!
    hashed = hash_password(data.password)
    user = User(email=data.email, password=hashed)
    db.add(user)
    await db.flush()
    return user
✅ With service layer
• Routes are thin — just parse + call + return
• Business logic is in testable service functions
• Services tested without FastAPI at all
• Services reused across multiple routes
• Each file stays small and focused
✅ THIN ROUTE — service does the work
# Route is just: receive → delegate → return
@router.post("/users", response_model=UserResponse)
async def create_user(
    data: UserCreate,
    db: DBSession,
):
    # ALL the logic is in the service
    return await user_service.create(db, data)


# Service has the real logic — easy to unit-test
class UserService:
    async def create(self, db: AsyncSession, data: UserCreate):
        if await self._email_exists(db, data.email):
            raise HTTPException(400, "Email taken")
        user = User(
            email=data.email,
            password=hash_password(data.password),
        )
        db.add(user)
        await db.flush()
        return user
Complete UserService — All CRUD Operations
The full service class covering get, list, create, update, and delete — the exact pattern used in production.
services/user_service.py
from sqlalchemy import select, func
from sqlalchemy.ext.asyncio import AsyncSession
from fastapi import HTTPException, status

class UserService:

    async def get(self, db: AsyncSession, user_id: int) -> User:
        user = await db.get(User, user_id)
        if not user:
            raise HTTPException(
                status_code=status.HTTP_404_NOT_FOUND,
                detail=f"User {user_id} not found",
            )
        return user

    async def list(
        self, db: AsyncSession,
        offset: int = 0, limit: int = 20,
    ):
        result = await db.execute(
            select(User)
            .where(User.is_active.is_(True))
            .order_by(User.created_at.desc())
            .offset(offset).limit(limit)
        )
        return result.scalars().all()

    async def create(
        self, db: AsyncSession, data: UserCreate
    ) -> User:
        if await self._email_exists(db, data.email):
            raise HTTPException(400, "Email already registered")
        user = User(**data.model_dump())
        db.add(user)
        await db.flush()  # populates user.id before commit
        return user

    async def update(
        self, db: AsyncSession,
        user_id: int, data: UserUpdate,
    ) -> User:
        user = await self.get(db, user_id)  # raises 404 if missing
        for field, value in data.model_dump(exclude_unset=True).items():
            setattr(user, field, value)
        await db.flush()
        return user

    async def delete(
        self, db: AsyncSession, user_id: int
    ) -> None:
        user = await self.get(db, user_id)
        await db.delete(user)
        await db.flush()

    async def _email_exists(
        self, db: AsyncSession, email: str
    ) -> bool:
        result = await db.scalar(
            select(func.count())
            .select_from(User)
            .where(User.email == email)
        )
        return (result or 0) > 0


# Singleton — import this instance everywhere
user_service = UserService()
routers/users.py — the thin routes
from fastapi import APIRouter, status
from app.core.db import DBSession
from app.services.user_service import user_service
from app.schemas.user import (
    UserCreate, UserUpdate, UserResponse, UserListResponse
)

router = APIRouter(prefix="/users", tags=["users"])


@router.get("/{user_id}", response_model=UserResponse)
async def get_user(user_id: int, db: DBSession):
    return await user_service.get(db, user_id)


@router.get("", response_model=UserListResponse)
async def list_users(
    db: DBSession,
    page: int = 1,
    page_size: int = 20,
):
    offset = (page - 1) * page_size
    items = await user_service.list(db, offset=offset, limit=page_size)
    return {"items": items, "page": page, "page_size": page_size}


@router.post("",
    response_model=UserResponse,
    status_code=status.HTTP_201_CREATED,
)
async def create_user(data: UserCreate, db: DBSession):
    return await user_service.create(db, data)


@router.patch("/{user_id}", response_model=UserResponse)
async def update_user(
    user_id: int, data: UserUpdate, db: DBSession
):
    return await user_service.update(db, user_id, data)


@router.delete("/{user_id}", status_code=204)
async def delete_user(user_id: int, db: DBSession):
    await user_service.delete(db, user_id)
Pydantic Schemas — Request and Response Shapes
Schemas validate incoming data and control what you expose in responses. Keep them separate from your SQLAlchemy models — they serve different purposes.
schemas/user.py — Pydantic schemas
from pydantic import BaseModel, EmailStr, ConfigDict
from datetime import datetime
from typing import List, Optional


# ── Incoming request to create a user ──
class UserCreate(BaseModel):
    name:     str
    email:    EmailStr
    password: str


# ── PATCH — all fields optional ──
class UserUpdate(BaseModel):
    name:  Optional[str] = None
    email: Optional[EmailStr] = None


# ── Single user in API response ──
class UserResponse(BaseModel):
    id:         int
    name:       str
    email:      str
    is_active:  bool
    created_at: datetime

    # allows reading from ORM model attributes
    model_config = ConfigDict(from_attributes=True)


# ── Paginated list response ──
class UserListResponse(BaseModel):
    items:     List[UserResponse]
    page:      int
    page_size: int
    total:     Optional[int] = None  # optional count
ORM model vs Pydantic schema — what goes where
Concern ORM Model Schema
DB table shape✅ Yes❌ No
Relationships✅ Yes❌ No
Request validation❌ No✅ Yes
API response shape❌ No✅ Yes
Hide sensitive fields❌ No✅ Yes
Alembic migrations✅ Yes❌ No
from_attributes=True

Add model_config = ConfigDict(from_attributes=True) to every response schema. Without it, UserResponse.model_validate(user_orm_object) fails because Pydantic can't read ORM object attributes by default.

Paginated Endpoints — Offset/Limit with Total Count

Production APIs always paginate lists. The standard pattern returns both the items for the current page and the total count so the client can render page numbers. Getting both in one round-trip requires running two queries — one for the data, one for the count.

SQLAlchemy makes this efficient. Both queries hit the same table and filters, but only the data query uses LIMIT/OFFSET. The count query uses func.count().

You can run both queries with await asyncio.gather() to fetch them in parallel — cutting latency in half on paginated endpoints.

Use query reuse for count

Build the base query once, then apply .limit().offset() for data and func.count() for the total. This keeps your filter logic in one place — if you add a new filter, both the data and the count update automatically.

PAGINATED LIST — data + total count
import asyncio
from sqlalchemy import select, func

class UserService:
    async def list_paginated(
        self,
        db: AsyncSession,
        page: int = 1,
        page_size: int = 20,
        search: str | None = None,
    ):
        # ── Build the base query with filters ──
        base_q = select(User).where(User.is_active.is_(True))
        if search:
            base_q = base_q.where(User.name.ilike(f"%{search}%"))

        # ── Two queries from the same base ──
        offset = (page - 1) * page_size
        data_q  = base_q.order_by(User.created_at.desc())\
                        .offset(offset).limit(page_size)
        count_q = select(func.count()).select_from(base_q.subquery())

        # ── Run both in parallel with asyncio.gather ──
        data_result, total = await asyncio.gather(
            db.execute(data_q),
            db.scalar(count_q),
        )
        items = data_result.scalars().all()

        return {
            "items":      items,
            "total":      total or 0,
            "page":       page,
            "page_size":  page_size,
            "pages":      (total + page_size - 1) // page_size,  # ceiling div
        }


# ── The route ──
@router.get("")
async def list_users(
    db: DBSession,
    page: int = 1,
    page_size: int = 20,
    search: str | None = None,
):
    return await user_service.list_paginated(
        db, page=page, page_size=page_size, search=search
    )

app/main.py — the FastAPI app entry point
from fastapi import FastAPI
from contextlib import asynccontextmanager
from app.core.db import engine
from app.models.base import Base
from app.routers import users


# ── Lifespan: startup + shutdown logic ──
@asynccontextmanager
async def lifespan(app: FastAPI):
    # startup — e.g. create tables in dev (use Alembic in prod)
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    yield
    # shutdown — dispose the connection pool
    await engine.dispose()


# ── Create the app ──
app = FastAPI(
    title="My API",
    lifespan=lifespan,
)

# ── Register routers ──
app.include_router(users.router, prefix="/api/v1")
Use lifespan, not on_event

@app.on_event("startup") is deprecated in FastAPI. Use the lifespan context manager instead — it runs startup code before yield and shutdown code after. This is the modern pattern and handles exceptions cleanly.
