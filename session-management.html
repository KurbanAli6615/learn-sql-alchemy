Understanding the Session Object

The Session — A Smart Buffer Between Python and the Database

The Session is an in-memory workspace. When you load objects, modify them, or create new ones, those changes are recorded in the session first. The session then decides when and how to send those changes to the database — either on the next flush(), or when you explicitly call commit().

This design is called the Unit of Work pattern. Instead of writing to the database immediately on every change, the session batches all pending changes into a single coordinated transaction. This is safer, faster, and easier to reason about.

The session also acts as an identity map — if you load the same row twice within a session, you get back the same Python object, not two copies. This prevents inconsistency where two parts of your code hold conflicting versions of the same row.

Session vs Connection vs Engine

Engine — knows how to connect to the database.
Connection — a raw database connection, one transaction at a time.
Session — the ORM layer on top. Tracks objects, manages the identity map, and wraps a connection/transaction for you.

CREATING AND USING A SESSION
from sqlalchemy import create_engine
from sqlalchemy.orm import Session, sessionmaker

engine = create_engine("postgresql://user:pass@localhost/db")


# ── Pattern 1: context manager (recommended) ──
with Session(engine) as session:
    user = User(name="Alice", email="alice@co")
    session.add(user)
    session.commit()
# session automatically closed here


# ── Pattern 2: sessionmaker factory ──
# Create the factory once at app startup
SessionLocal = sessionmaker(bind=engine)

# Then use it throughout the app
with SessionLocal() as session:
    users = session.execute(
        select(User)
    ).scalars().all()


# ── Pattern 3: FastAPI / Flask dependency ──
def get_db():
    db = SessionLocal()
    try:
        yield db
        db.commit()
    except Exception:
        db.rollback()
        raise
    finally:
        db.close()

flush() vs commit() — The Most Confusing Distinction
Both send SQL to the database, but they are not the same. Confusing them is the #1 source of "where did my data go?" bugs.
flush() — sends SQL, stays in transaction
1
Translates all pending Python changes (INSERT, UPDATE, DELETE) into SQL statements and sends them to the database.
2
The database executes the SQL inside the current transaction — not yet committed. Other connections cannot see these changes.
3
The session remains open. You can still add more changes, or call rollback() to undo everything flushed so far.
4
Auto-flush happens before every query so your queries see the latest state of objects you've already added.
│
commit() — flush + make permanent
1
Calls flush() first — all pending changes are sent to the database.
2
Commits the transaction to the database. Changes are now permanent and visible to all other connections.
3
The session expires all objects — their attributes will be refreshed from the database on next access to ensure you see fresh data.
4
Cannot be undone after commit. If you need to undo, call rollback() before committing.

INSPECTING OBJECT STATES
from sqlalchemy import inspect

# TRANSIENT — not in any session
user = User(name="Alice")
print(inspect(user).transient)   # True

with Session(engine) as session:

    # PENDING — added, not yet flushed
    session.add(user)
    print(inspect(user).pending)    # True
    print(user.id)                  # None — no PK yet

    # PERSISTENT — after flush or commit
    session.flush()
    print(inspect(user).persistent) # True
    print(user.id)                  # 1 — PK assigned

    session.commit()
    print(inspect(user).persistent) # True

# DETACHED — session is closed
print(inspect(user).detached)    # True
print(user.id)                    # 1 — PK still accessible
# user.name — OK, not a lazy relationship
# user.orders — ❌ DetachedInstanceError!
The identity map in action
Loading the same row twice returns the same Python object — changes are consistent.
IDENTITY MAP — SAME OBJECT GUARANTEE
with Session(engine) as session:

    # Load user with id=1 twice
    user_a = session.get(User, 1)
    user_b = session.get(User, 1)

    # Same object — no second SQL query!
    print(user_a is user_b)          # True
    print(id(user_a) == id(user_b))  # True

    # Modify via one reference…
    user_a.name = "Alice Updated"

    # …reflected in the other
    print(user_b.name)  # "Alice Updated"

    # Expire an object — forces refresh on next access
    session.expire(user_a)
    print(user_a.name)  # re-fetches from DB

    # Refresh now (not lazily)
    session.refresh(user_a)
    print(user_a.name)  # fresh from DB
Why the identity map matters

Without it, two parts of your code could load the same user row, modify different attributes, and then commit conflicting states. The identity map guarantees you always work with the same in-memory object.

Transactions — All-or-Nothing Database Operations

A transaction groups multiple SQL statements into a single atomic unit. Either all statements succeed and are committed, or the entire group is rolled back as if none of them happened. This is the A in ACID — Atomicity.

SQLAlchemy begins a transaction automatically the first time you use a session. You don't write BEGIN explicitly. You close the transaction by calling commit() (persist changes) or rollback() (discard changes). After either, the session is ready for the next transaction.

When an exception is raised inside a with Session() block, the context manager calls rollback() automatically — you never accidentally leave a half-committed transaction open.

ACID — the four transaction guarantees

Atomic — all or nothing.
Consistent — DB moves from one valid state to another.
Isolated — concurrent transactions don't interfere (see Part 3).
Durable — committed data survives crashes.

TRANSACTION PATTERNS
# ── Basic commit and rollback ──
with Session(engine) as session:
    try:
        user = User(name="Alice")
        session.add(user)

        order = Order(user_id=user.id, product="Laptop")
        session.add(order)

        session.commit()   # both saved or neither
    except Exception:
        session.rollback() # undo EVERYTHING since last commit
        raise


# ── Context manager handles rollback automatically ──
with Session(engine) as session:
    with session.begin():        # auto-commits or rolls back
        session.add(User(name="Bob"))
        session.add(User(name="Carol"))
    # commit() called automatically if no exception
    # rollback() called automatically on exception


# ── Check transaction state ──
with Session(engine) as session:
    session.add(User(name="Dave"))

    # True — active transaction with pending changes
    print(session.dirty)         # set of modified objects
    print(session.new)           # set of pending objects
    print(session.deleted)       # set of deleted objects
    print(session.is_active)     # True

SAVEPOINTS IN SQLAlchemy
with Session(engine) as session:

    # Start the outer transaction
    user = User(name="Alice")
    session.add(user)
    session.flush()        # user gets its PK

    # Create a savepoint — point to roll back to
    with session.begin_nested() as sp:
        try:
            order = Order(user_id=user.id, amount=0)
            session.add(order)
            session.flush()    # will fail (CHECK constraint)
        except Exception:
            sp.rollback()     # only the order is rolled back
            print("Order failed, user kept")

    # Outer transaction still active — user is intact
    session.commit()         # commits the user, not the order
BULK OPERATIONS IN A TRANSACTION
from sqlalchemy import select

# Reading and writing in the same transaction
with Session(engine) as session:

    # Add multiple objects in one go
    session.add_all([
        User(name="Alice", email="a@co"),
        User(name="Bob",   email="b@co"),
        User(name="Carol", email="c@co"),
    ])
    session.flush()            # send INSERTs, stay in txn

    # Count is visible within this transaction
    count = session.execute(
        select(func.count()).select_from(User)
    ).scalar()
    print(f"{count} users in session")

    # Merge (upsert) — insert or update by PK
    updated = session.merge(
        User(id=1, name="Alice Updated")
    )
    session.commit()            # all at once
session.merge() — the upsert shortcut

Pass an object with a primary key. If a row with that PK exists, it is updated. If not, it is inserted. This is the ORM equivalent of INSERT ... ON CONFLICT DO UPDATE.

2B
Bulk Operations
Updating or deleting thousands of rows one object at a time is slow. One SQL statement beats a Python loop every time.
BULK OPS

Bulk Update, Delete & Insert — One Statement, Not a Loop

The ORM's default workflow is object-by-object: load a list of objects, change each one, commit. This is clear and readable for a few rows, but at scale it generates one SQL statement per object. 1000 users = 1000 UPDATE statements.

Bulk operations send a single SQL statement to the database regardless of how many rows are affected. The database is specifically designed to handle this efficiently — a single UPDATE ... WHERE id IN (…) is orders of magnitude faster than a loop.

The trade-off: bulk operations bypass the ORM's identity map and don't update Python objects that are already loaded in the session. After a bulk update, call session.expire_all() to make SQLAlchemy reload fresh values on next access.

The loop trap

The most common performance mistake in SQLAlchemy: fetching 10,000 rows with scalars().all() and then looping through them updating each one. This generates 10,001 SQL statements. Use a bulk update instead — it's one statement.

BULK UPDATE & DELETE
from sqlalchemy import update, delete, select

# ── Bulk UPDATE — set status for a list of IDs ──
user_ids = [1, 2, 3, 4, 5]
session.execute(
    update(User)
    .where(User.id.in_(user_ids))
    .values(status="active")
)
session.commit()
# → UPDATE users SET status='active' WHERE id IN (1,2,3,4,5)
# → ONE query, not five


# ── Bulk UPDATE with a WHERE condition ──
session.execute(
    update(User)
    .where(User.last_login < "2023-01-01")
    .values(status="inactive")
)
session.commit()


# ── Bulk DELETE ──
session.execute(
    delete(Session)
    .where(Session.expired_at <= "2024-01-01")
)
session.commit()
# → DELETE FROM sessions WHERE expired_at <= '2024-01-01'


# ── After bulk ops: refresh in-memory objects ──
session.expire_all()
# Forces SQLAlchemy to reload from DB on next access
BULK INSERT — many rows at once
from sqlalchemy import insert

# ── Bulk INSERT — insert many rows with one statement ──
session.execute(
    insert(Product),
    [
        {"name": "Widget A", "price": 9.99},
        {"name": "Widget B", "price": 14.99},
        {"name": "Widget C", "price": 4.99},

# ── UPSERT — insert, or update if already exists ──
from sqlalchemy.dialects.postgresql import insert as pg_insert

stmt = pg_insert(Product).values(
    id=42, name="Widget A", price=9.99

stmt = stmt.on_conflict_do_update(
    index_elements=["id"],
    set_={"price": stmt.excluded.price}

Isolation Levels — What Concurrent Transactions Can See

When two transactions run at the same time, the database must decide whether Transaction A can see uncommitted changes from Transaction B. This is controlled by the isolation level.

Higher isolation means stronger consistency guarantees but lower concurrency — the database must do more locking. Lower isolation allows more parallelism but risks concurrency anomalies: dirty reads, non-repeatable reads, and phantom reads.

Most applications work well with READ COMMITTED (the default in PostgreSQL) or REPEATABLE READ for critical financial operations. SERIALIZABLE is the safest but slowest.

The three concurrency problems

Dirty Read — reading uncommitted data from another transaction.
Non-Repeatable Read — same row reads differently twice (another txn updated it between reads).
Phantom Read — same query returns different rows twice (another txn inserted/deleted rows).

SETTING ISOLATION LEVELS
from sqlalchemy import create_engine, text

# ── Engine-level isolation (default for all sessions) ──
engine = create_engine(
    "postgresql://user:pass@localhost/db",
    isolation_level="REPEATABLE READ"
)


# ── Connection-level (override for one operation) ──
with engine.connect() as conn:
    conn = conn.execution_options(
        isolation_level="SERIALIZABLE"
    )
    conn.execute(...)
    conn.commit()


# ── Available isolation level strings ──
# "READ UNCOMMITTED"  — lowest,  fastest
# "READ COMMITTED"    — default in PostgreSQL
# "REPEATABLE READ"   — default in MySQL InnoDB
# "SERIALIZABLE"      — highest, safest, slowest
# "AUTOCOMMIT"        — no transaction wrapper

from sqlalchemy import Integer, Column

class Account(Base):

    version = Column(Integer, nullable=False)

Optimistic vs Pessimistic Locking — Two Approaches to Concurrent Writes
When two users try to update the same row simultaneously, one of them must lose. How they lose is determined by your locking strategy.
🔓
Optimistic Locking
Assume conflicts are rare. Let both transactions proceed. When committing, check if anyone else changed the row (using a version or updated_at column). If changed, raise an error. Best for high-read, low-conflict workloads (product listings, user profiles).

SQLAlchemy support: __mapper_args__ = {"version_id_col": version} — auto-increments on UPDATE, raises StaleDataError on conflict.
🔒
Pessimistic Locking
Assume conflicts are likely. Lock the row when you read it so no other transaction can modify it. The second reader blocks until the first commits. Best for inventory counts, bank balances — data that must be strictly serialised.

SQLAlchemy support: .with_for_update() adds SELECT ... FOR UPDATE. The locked row cannot be updated by other transactions until this one commits.
📋
SELECT FOR UPDATE
Acquires a write lock on the rows returned by the query. Other transactions trying to update those rows will block. The lock is released on COMMIT or ROLLBACK.

session.execute(select(Account).where(Account.id == 1).with_for_update()).scalar_one()
⚠️
StaleDataError
Raised by SQLAlchemy's optimistic locking when the version number doesn't match on commit. This means another transaction modified the row between your read and your write. Handle it by retrying the operation with fresh data from the database.

except StaleDataError: session.rollback(); retry()
OPTIMISTIC LOCKING — version_id_col
from sqlalchemy.orm import Mapped, mapped_column
from sqlalchemy.exc import StaleDataError

class Account(Base):
    __tablename__ = "accounts"
    id:      Mapped[int] = mapped_column(primary_key=True)
    balance: Mapped[int]
    version: Mapped[int] = mapped_column(default=1)

    __mapper_args__ = {"version_id_col": version}

# When two sessions update the same account:
session_a = Session(engine)
session_b = Session(engine)

acct_a = session_a.get(Account, 1)  # version=1
acct_b = session_b.get(Account, 1)  # version=1

acct_a.balance += 100
session_a.commit()              # version bumped to 2

acct_b.balance += 50
try:
    session_b.commit()           # version mismatch → error!
except StaleDataError:
    session_b.rollback()
    print("Conflict detected — retry")
PESSIMISTIC LOCKING — with_for_update()
from sqlalchemy import select

# Lock the row for the duration of the transaction
with Session(engine) as session:
    with session.begin():

        # SELECT balance WHERE id=1 FOR UPDATE
        acct = session.execute(
            select(Account)
            .where(Account.id == 1)
            .with_for_update()
        ).scalar_one()

        # Row is now locked — other sessions block here
        acct.balance += 100

    # Lock released on commit
    # Other sessions that were blocked can now proceed


# SKIP LOCKED — skip rows already locked by others
# (useful for job queues)
result = session.execute(
    select(Job)
    .where(Job.status == "pending")
    .limit(1)
    .with_for_update(skip_locked=True)
)
skip_locked for job queues

In worker pools, each worker takes a job and locks it. skip_locked=True makes other workers skip already-locked jobs instead of blocking — they move on to the next available one. This prevents the thundering herd problem.
