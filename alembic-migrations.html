The Problem — Databases Change, But How Do You Track That?

When you build an app, your database tables change all the time. You add a new column, rename one, add an index. In code, you just edit a file and commit it to Git. But your database is different — it lives on a server, and there is no automatic history of what changed.

Imagine two developers working on the same project. Developer A adds a phone_number column on their laptop. Developer B doesn't have it. Your production server has yet another version. Nobody knows which version is "correct."

Alembic fixes this. Every schema change becomes a numbered migration file that lives in your Git repository alongside your code. Anyone who pulls your code can run one command and their database is up to date.

Simple analogy

Think of Alembic exactly like Git. Git tracks changes to your code files. Alembic tracks changes to your database tables. Both give you a full history, and both let you go back to any previous version.

Git vs Alembic — same idea, different target
📁
Git
Tracks changes to code files. Every commit is a snapshot. You can checkout any old version.
🗄️
Alembic
Tracks changes to database tables. Every migration is a snapshot. You can roll back to any version.
git commit -m "add phone field"
git checkout v1.2
alembic upgrade head
alembic downgrade -1

Where Alembic Sits in Your Stack
Alembic is the layer between your Python models and the database. It reads your models, detects what changed, and writes the SQL to update the database safely.
Technology stack — who does what
🐍
Your Python Code
You define models using SQLAlchemy classes. When you add a new column, you add it here first.
▼
🔄
Alembic
Compares your models to the current database, generates migration files, and runs them in order.
▼
🔗
SQLAlchemy Core
Translates migration operations (add_column, create_table) into the correct SQL for your database.
▼
🗄️
Your Database
PostgreSQL, MySQL, or SQLite. Alembic creates a special alembic_version table here to track which migrations have run.
The alembic_version table — Alembic's memory
alembic_version table (auto-created by Alembic)
version_num
3d4e5f6a7b8c
This single-row table stores the ID of the last migration that ran. Every time you run alembic upgrade, Alembic checks this table, finds which migrations haven't run yet, and runs them in order.
Alembic is a separate package

Alembic is not part of SQLAlchemy — it's a separate library by the same author. Install it with pip install alembic. It works alongside SQLAlchemy but doesn't require it (though it's almost always used together).

Setting Up Alembic — Four Commands to Get Started

Getting Alembic running takes four steps. You install it, tell it about your database, point it to your models, and you're ready to create your first migration. The whole setup takes about five minutes.

The most important file is env.py — this is where you connect Alembic to your SQLAlchemy models so it can auto-detect schema changes. Once this is set up, you never touch it again.

Run these commands once

You only run alembic init alembic once per project to create the folder structure. After that, you use alembic revision and alembic upgrade for every change.

SETUP — 4 STEPS
# Step 1: Install Alembic
pip install alembic


# Step 2: Initialise Alembic in your project
# This creates the alembic/ folder + alembic.ini
alembic init alembic


# Step 3: Edit alembic.ini — set your database URL
# sqlalchemy.url = postgresql://user:pass@localhost/mydb


# Step 4: Edit alembic/env.py — point to your models
from myapp.models import Base
target_metadata = Base.metadata
# This lets Alembic auto-detect your model changes

Key section of env.py — connect Alembic to your models
alembic/env.py — the important part
from myapp.database import engine
from myapp.models import Base

# Tell Alembic about your models
# This is what makes auto-generate work
target_metadata = Base.metadata

def run_migrations_online():
    connectable = engine
    with connectable.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=target_metadata
        )
        with context.begin_transaction():
            context.run_migrations()
The key line

target_metadata = Base.metadata — this tells Alembic "here are all my models." Without this, auto-generate won't work and Alembic can't detect your schema changes.

Inside a Migration File — Every Part Explained
Every migration file has the same structure. There are only two things you need to understand: upgrade() applies the change, downgrade() reverses it.
Anatomy of a migration file — alembic/versions/002_add_phone.py
002_add_phone_number.py
"""add phone number column
Revision ID: a1b2c3d4e5f6
Revises:     9z8y7x6w5v4u
Create Date: 2024-03-15 10:30:00
"""
from alembic import op
import sqlalchemy as sa

# This migration's unique ID
revision = 'a1b2c3d4e5f6'

# ID of the migration that comes before this one
down_revision = '9z8y7x6w5v4u'

# These stay the same in every migration file
branch_labels = None
depends_on = None


def upgrade():
    # What to do going FORWARD
    op.add_column(
        'users',
        sa.Column('phone_number', sa.String(20), nullable=True)
    )


def downgrade():
    # What to do going BACKWARD (undo upgrade)
    op.drop_column('users', 'phone_number')
REVISION ID
A unique hash ID for this migration. Alembic stores this in the alembic_version table to know it has run.
DOWN_REVISION
The ID of the migration that came before this one. This is how the chain of migrations is linked — like a linked list.
UPGRADE()
What to do when moving forward. Called by alembic upgrade. This is where you add columns, create tables, etc.
DOWNGRADE()
What to do when moving backward. Called by alembic downgrade. Always the exact opposite of upgrade().

Every migration points to the one before it using down_revision. This creates a chain. Alembic walks this chain to know which order to run migrations and how far back to roll.

alembic upgrade head
Runs all missing migrations up to the latest (HEAD). Safe to run anytime — skips ones already run.
alembic upgrade +2
Runs exactly 2 migrations from the current position. Useful for step-by-step testing.
alembic downgrade -1
Rolls back the last migration. Calls its downgrade() function and reverts the DB change.

Writing Real Migrations — Common Operations With Code
These are the operations you'll use in 90% of your migrations. Every one is reversible.
CREATING AND CHANGING TABLES
from alembic import op
import sqlalchemy as sa

def upgrade():

    # Create a brand-new table
    op.create_table(
        'products',
        sa.Column('id',    sa.Integer(), primary_key=True),
        sa.Column('name',  sa.String(100), nullable=False),
        sa.Column('price', sa.Numeric(10, 2)),
    )

    # Add a new column to an existing table
    op.add_column(
        'users',
        sa.Column('bio', sa.Text(), nullable=True)
    )

    # Rename a column
    op.alter_column(
        'users', 'username',
        new_column_name='display_name'
    )

    # Add a database index
    op.create_index(
        'ix_users_email',
        'users', ['email']
    )

def downgrade():
    op.drop_index('ix_users_email', table_name='users')
    op.alter_column('users', 'display_name', new_column_name='username')
    op.drop_column('users', 'bio')
    op.drop_table('products')
DATA MIGRATIONS — MOVING DATA
# Sometimes you need to move data, not just schema
from alembic import op
import sqlalchemy as sa

def upgrade():
    # Step 1: Add new column (nullable first)
    op.add_column(
        'users',
        sa.Column('full_name', sa.String(), nullable=True)
    )

    # Step 2: Populate data from old columns
    conn = op.get_bind()
    conn.execute(sa.text(
        """UPDATE users
           SET full_name = first_name || ' ' || last_name"""
    ))

    # Step 3: Now make it NOT NULL
    op.alter_column('users', 'full_name',
                    nullable=False)

    # Step 4: Drop the old columns
    op.drop_column('users', 'first_name')
    op.drop_column('users', 'last_name')

def downgrade():
    op.add_column('users', sa.Column('first_name', sa.String()))
    op.add_column('users', sa.Column('last_name',  sa.String()))
    op.drop_column('users', 'full_name')

Auto-Generate vs Manual — When to Use Which

Alembic can automatically generate migration files by comparing your Python models to the current database. This saves time — you change your model, run one command, and Alembic writes the migration for you.

But auto-generate isn't perfect. It can miss some changes (like changing a column's default value or a stored procedure). And it never writes your downgrade() logic for data migrations. Always review the generated file before running it.

Always review auto-generated migrations

Auto-generate compares table structures. It cannot read your mind about data. If you rename a column, it may generate DROP + ADD instead of ALTER. Review the file before running it in production.

Auto-generate vs manual — choose the right tool
✨ Auto-generate
✓Adding new columns
✓Creating new tables
✓Adding indexes
✓Changing column types
✗Data migrations
✗Stored procedures
alembic revision --autogenerate -m "msg"
✍ Write Manually
✓Moving / copying data
✓Seeding default values
✓Complex constraints
✓Stored procedures / views
✓Splitting / merging tables
alembic revision -m "seed users"
Team Workflow — How Every Developer Stays in Sync
This is the process every developer should follow. It's the same 4-step pattern every time, whether you're adding a column or deploying to production.
Standard team migration workflow
1
Change your SQLAlchemy model
Add the new column, table, or relationship in your Python model class. Don't touch the database yet.
class User(Base):
  phone = Column(String)
2
Auto-generate the migration file
Alembic compares your model to the database, detects the new column, and writes the migration file for you.
alembic revision --autogenerate -m "add phone"
3
Review and commit the file
Open the generated file, check that upgrade() and downgrade() look correct, then commit both your model change and the migration file together in one Git commit.
git add models.py alembic/versions/xxx.py
git commit -m "add phone number column"
4
Run upgrade on every environment
After git pull, every developer (and your deployment pipeline) runs this one command to bring their database up to the latest version.
alembic upgrade head

Best Practices — The Rules That Prevent Production Disasters
Follow these rules every time. They exist because someone, somewhere, broke them and lost data. Learn from their mistakes.
✅ Always do these
✅
Always write a downgrade() function
Every upgrade must have a matching downgrade. If you can't roll back a migration, you can't fix it when it breaks in production. No exceptions.
✅
Commit migrations with the code that needs them
Always commit your model change and its migration file in the same Git commit. This keeps code and database in sync when your teammates pull.
✅
Add columns as nullable first
When adding a NOT NULL column to a table with existing rows, add it as nullable first, fill in data, then make it NOT NULL. Otherwise the migration fails on non-empty tables.
✅
Test migrations on a copy of production data
Before running a migration in production, run it against a backup copy of the production database. Schema changes that work on 10 rows can fail on 10 million rows.
❌ Never do these
❌
Never edit an old migration file
Once a migration has run on any environment (dev, staging, production), it is locked. Editing it means other environments will never see the change. Create a new migration instead.
❌
Never change the database manually in production
If you manually ALTER TABLE in production, Alembic doesn't know about it. The next migration may fail or produce wrong results. All changes go through Alembic — always.
❌
Never run alembic downgrade base in production
downgrade base drops everything. This is only for local development. In production, roll back exactly one migration at a time with downgrade -1.
❌
Never skip reviewing auto-generated files
Auto-generate is helpful but not perfect. Renaming a column may generate a DROP + ADD (which loses data!) instead of an ALTER. Always read the generated file before running it.
