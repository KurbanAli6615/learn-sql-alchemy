01 / 12

What is SQLAlchemy Core — and why learn it?

Think of SQLAlchemy as a toolbox. Inside that toolbox there are two layers:

The ORM (Object-Relational Mapper) is the top layer — it lets you work with Python class objects and forgets about SQL completely. You write session.query(User).all() and SQLAlchemy figures out the SQL for you.

The Core is the bottom layer — it is closer to SQL. You still write Python code, but you are much more aware of the actual SQL being produced. You build statements like select(users).where(users.c.age > 25) and Core turns that into clean SQL.

Core is not a replacement for the ORM — they work together. The ORM is built on top of Core. Understanding Core makes you a much better SQLAlchemy developer because you understand exactly what is happening underneath.

Simple analogy

The ORM is like driving a car with automatic transmission — easy, comfortable, handles things for you. Core is like driving with a manual transmission — more control, you decide every gear shift. Both get you to the same destination.

6 Reasons to Learn Core
Why bother with Core when the ORM exists?
🔍
Understand what the ORM is doing
Every ORM query eventually becomes Core. When you understand Core, you can read the generated SQL, spot slow queries, and fix them — instead of being confused by what SQLAlchemy is doing behind the scenes.
⚡
Performance-critical queries
For bulk inserts, complex joins with many tables, or analytics queries over millions of rows, Core gives you fine-grained control over the exact SQL. No hidden lazy loads, no unexpected extra queries.
🧩
Migrations & schema tools
Alembic (the migration tool) is built on Core. When you write migration files to add columns, create indexes, or drop tables, you are writing Core code. You need Core to use Alembic properly.
🔄
Works without models
Sometimes you are querying a legacy database or a table you did not create. With Core, you can reflect existing tables and query them without defining any ORM models at all.
🎯
Database-agnostic SQL
Core automatically adjusts syntax for your database. The same Python code runs on PostgreSQL, MySQL, and SQLite — Core handles the dialect differences (quoting, data types, LIMIT syntax) for you.
📚
Foundation for everything
Once you know Core, the ORM becomes much easier to understand. All advanced ORM features — hybrid properties, custom types, event listeners — rely on Core concepts. Learn it once, benefit everywhere.

The Expression Language — Python that thinks in SQL

The SQL Expression Language is the heart of SQLAlchemy Core. It is a set of Python objects and functions that represent SQL constructs — tables, columns, conditions, and full queries.

Instead of writing a raw SQL string like "SELECT name FROM users WHERE age > 25", you write Python code like select(users.c.name).where(users.c.age > 25). SQLAlchemy then compiles that Python object into the correct SQL string for your database.

This is the key idea: you describe what you want in Python, and SQLAlchemy writes the SQL for you — but unlike the ORM, you are very close to the SQL. There are no hidden queries, no surprises.

Python Expression → Compiled SQL
Python (SQLAlchemy Expression)
# "Give me all users"
select(users)
→
SQL Generated
SELECT users.id, users.name,
       users.email, users.age, users.city
FROM users
Python (SQLAlchemy Expression)
# "Users older than 25, sorted"
select(users)
  .where(users.c.age > 25)
  .order_by(users.c.name)
→
SQL Generated
SELECT users.id, users.name,
       users.email, users.age, users.city
FROM users
WHERE users.age > 25
ORDER BY users.name
Python (SQLAlchemy Expression)
# "Count users per city"
select(
  users.c.city,
  func.count(users.c.id)
).group_by(users.c.city)
→
SQL Generated
SELECT users.city,
       count(users.id) AS count_1
FROM users
GROUP BY users.city
The .c accessor — how you reference columns

When you define a table object (we'll do this in the next section), every column in that table is accessible through the .c accessor (short for ".columns").

So if you have a table called users with a column called name, you reference that column in expressions as users.c.name. This gives you a Column object — a Python object that knows it represents the name column in the users table.

When you write users.c.age > 25 in Python, SQLAlchemy does not compare anything right now. It creates a condition expression — a Python object that means "age is greater than 25". This expression is only turned into SQL when you execute it.

Why not just write a string?

Writing users.c.age > 25 instead of "age > 25" means SQLAlchemy can check column names, handle different database quoting styles, safely parametrize values (preventing SQL injection), and produce the exact right SQL for your database.

EXPRESSIONS — PYTHON OBJECTS, NOT STRINGS
from sqlalchemy import (
    Table, Column, Integer, String,
    MetaData, select, func
)

# A column reference — this is a Python object
col = users.c.name
# col → Column('name', String(), table=<users>)

# A condition expression — still a Python object
condition = users.c.age > 25
# condition → BinaryExpression(age > :age_1)

# A full SELECT statement — still a Python object
stmt = select(users).where(condition)
# stmt → Select object, not yet executed

# Only NOW does it hit the database
with engine.connect() as conn:
    result = conn.execute(stmt)
    # ↑ COMPILES stmt to SQL, sends it, gets rows

# Print the SQL string (for debugging)
print(str(stmt))
# → SELECT users.id, users.name, ...
#   FROM users WHERE users.age > :age_1

Working with Tables & Schemas

In SQLAlchemy Core, your database schema is described using three objects that work together: MetaData, Table, and Column.

MetaData is like a big container — it holds all your table definitions. Think of it as a registry or a dictionary that knows about every table in your database.

Table represents one database table. It is given a name (like "users") and is registered inside MetaData. It holds the column definitions.

Column describes one column inside a table — its name, its data type, and any constraints (like primary key, not null, unique).

Defining the schema in Python

This is how you define both tables above using SQLAlchemy Core. First you create a MetaData object (one per project, created once). Then you create Table objects, each containing Column objects.

Notice how ForeignKey("users.id") creates the relationship between the two tables — this is just like a real database foreign key constraint.

When you call metadata.create_all(engine) at the bottom, SQLAlchemy sends CREATE TABLE SQL statements to your database and creates the tables. If the tables already exist, nothing happens (safe to call multiple times).

MetaData is the registry

You only create one MetaData object for your whole project. Every Table you define is automatically registered inside it. When you call metadata.create_all(engine), it creates ALL registered tables at once.

schema.py — defining tables with Core
from sqlalchemy import (
    create_engine, MetaData, Table,
    Column, Integer, String, Float,
    ForeignKey
)

# 1. Create the engine (connection to database)
engine = create_engine("sqlite:///shop.db")

# 2. Create ONE MetaData object for the whole project
metadata = MetaData()

# 3. Define the 'users' table
users = Table(
    "users", metadata,
    Column("id",    Integer, primary_key=True),
    Column("name",  String(100), nullable=False),
    Column("email", String(200), unique=True),
    Column("age",   Integer),
    Column("city",  String(100)),
)

# 4. Define the 'orders' table
orders = Table(
    "orders", metadata,
    Column("id",      Integer, primary_key=True),
    Column("user_id", Integer, ForeignKey("users.id")),
    Column("product", String(200)),
    Column("amount",  Float),
    Column("status",  String(50)),
)

# 5. Create ALL tables in the database at once
metadata.create_all(engine)
# → sends CREATE TABLE SQL to the database
Column Types Reference
The most common Column types you will use
SQLAlchemy maps these Python types to the correct SQL type for your database automatically.
Integer
A whole number, no decimal point. Used for IDs, counts, ages, quantities.
→ SQL: INTEGER  ·  Python: int
String(n)
Text with a maximum length of n characters. Used for names, emails, titles, status values. Always pass a length.
→ SQL: VARCHAR(n)  ·  Python: str
Text
Unlimited-length text. Used for descriptions, blog posts, comments. No length limit needed.
→ SQL: TEXT  ·  Python: str
Float
A decimal number. Used for prices, measurements, percentages. For money use Numeric instead (avoids floating point errors).
→ SQL: FLOAT  ·  Python: float
Numeric(p,s)
Exact decimal. p = total digits, s = decimal places. Use this for money to avoid rounding issues.
Numeric(10,2) → 99999999.99
Boolean
True or False. Used for flags, toggles, is_active, is_admin, etc.
→ SQL: BOOLEAN  ·  Python: bool
DateTime
A date and time value. Used for created_at, updated_at, timestamps.
→ SQL: DATETIME  ·  Python: datetime
JSON
Stores a JSON object or array. Useful for flexible/semi-structured data. Supported in PostgreSQL natively.
→ SQL: JSON / JSONB  ·  Python: dict/list
Three important MetaData methods
metadata.create_all(engine)
Creates all tables that are registered in this MetaData object. Safe to call multiple times — it only creates tables that do not exist yet. Use at app startup.
metadata.drop_all(engine)
Drops (deletes) all tables registered in this MetaData. This deletes all data. Only use in development or tests, never in production without a backup.
autoload_with=engine
Reads an existing table from the database and creates a Table object from it automatically. Useful for legacy databases you did not design.
REFLECTING AN EXISTING TABLE
# If the table already exists in the database, you can "reflect" it
# instead of redefining all the columns manually
metadata = MetaData()
existing_table = Table("orders", metadata, autoload_with=engine)

# SQLAlchemy reads the column names, types, and constraints from the database
# You can now use existing_table.c.id, existing_table.c.product, etc.
print([col.name for col in existing_table.c])
# → ['id', 'user_id', 'product', 'amount', 'status']

SELECT — Reading Data from a Table

SELECT

SELECT is how you read data from a database. It does not change anything — it just retrieves rows. It is the most used SQL operation by far.

In SQLAlchemy Core, you use the select() function to build a SELECT statement. You pass in the table or specific columns you want, then optionally add .where() to filter rows.

4.1 — Select ALL rows and ALL columns
The simplest SELECT: give me everything from the users table. Pass the entire table object to select(). SQLAlchemy expands it into all columns automatically.
Python
from sqlalchemy import select

stmt = select(users)

with engine.connect() as conn:
    result = conn.execute(stmt)
    rows = result.fetchall()
    for row in rows:
        print(row.name, row.age)
→

4.3 — Select with WHERE — filtering rows
Add .where() to filter which rows you get back. Only rows that match the condition are returned. Here we get only users older than 27.
Python
stmt = select(users).where(
    users.c.age > 27
)

with engine.connect() as conn:
    result = conn.execute(stmt)
    rows = result.fetchall()
    # rows → Alice(28), Bob(34), Diana(31)
→
SQL Generated
SELECT users.id, users.name,
       users.email, users.age, users.city
FROM users
WHERE users.age > 27
Values are parametrized automatically

Notice the SQL shows :age_1 internally (a named parameter), not the raw value 27 hardcoded into the string. SQLAlchemy always uses bound parameters — this is what prevents SQL injection attacks. You never need to escape values manually.

4.4 — Reading results — fetchall, fetchone, scalar
RESULT OBJECT METHODS
with engine.connect() as conn:
    result = conn.execute(select(users))

    # fetchall() → returns ALL rows as a list of Row objects
    all_rows = result.fetchall()
    # → [(1, 'Alice', ...), (2, 'Bob', ...), ...]

    # fetchone() → returns the FIRST row only, then advances cursor
    one_row = result.fetchone()
    # → (1, 'Alice', 'alice@example.com', 28, 'New York')

    # Access columns by name (like a dictionary)
    print(one_row.name)   # → 'Alice'
    print(one_row.age)    # → 28
    print(one_row[1])     # → 'Alice' (by index too)

    # scalars() → iterate and return first column of each row
    names = conn.execute(select(users.c.name)).scalars().all()
    # → ['Alice', 'Bob', 'Charlie', 'Diana', 'Eve']

    # mappings() → rows as dicts  {column_name: value}
    dicts = conn.execute(select(users)).mappings().all()
    # → [{'id':1,'name':'Alice',...}, {'id':2,'name':'Bob',...}]

INSERT — Adding New Rows

INSERT

INSERT is how you add new rows to a table. In SQLAlchemy Core, you use insert(table).values(...) to build the statement. You must then execute it inside a connection that can commit — use engine.begin() so changes are saved automatically.

This is the key difference: engine.connect() is for reading (SELECT), engine.begin() is for writing (INSERT, UPDATE, DELETE). If you use engine.connect() for writes, you must call conn.commit() manually.

5.1 — Insert a single row
Use insert(table).values(column=value, ...). Wrap in engine.begin() — this opens a transaction and automatically commits when the block ends, or rolls back if an error happens.
Python
from sqlalchemy import insert

stmt = insert(users).values(
    name="Frank",
    email="frank@example.com",
    age=29,
    city="Sydney"
)

# engine.begin() → auto-commit on success
with engine.begin() as conn:
    result = conn.execute(stmt)
    print(result.inserted_primary_key)
    # → (6,) — the new id
→

To insert many rows efficiently, pass a list of dictionaries to conn.execute(). This sends a single INSERT with multiple value sets — much faster than inserting one row at a time in a loop.

UPDATE — Changing Existing Rows

UPDATE

UPDATE changes the value of one or more columns in rows that match a condition. In SQLAlchemy Core, you use update(table).where(condition).values(column=new_value).

Always use .where() with UPDATE — without it, you update every single row in the table. That is almost never what you want.

6.1 — Update a single column for one row
Change Bob's city from "London" to "Amsterdam". The .where() narrows it down to Bob only (id = 2). The .values() sets the new value.
Python
from sqlalchemy import update

stmt = (
    update(users)
    .where(users.c.id == 2)
    .values(city="Amsterdam")
)

with engine.begin() as conn:
    result = conn.execute(stmt)
    print(result.rowcount)
    # → 1  (one row was changed)

DELETE — Removing Rows

DELETE

DELETE removes rows from a table permanently. In SQLAlchemy Core, you use delete(table).where(condition). Like UPDATE, always include a WHERE clause — without it, you delete every row in the table.

7.1 — Delete one specific row
Remove the user whose id is 3 (Charlie). The WHERE clause makes sure only that one row is deleted.
Python
from sqlalchemy import delete

stmt = (
    delete(users)
    .where(users.c.id == 3)
)

with engine.begin() as conn:
    result = conn.execute(stmt)
    print(result.rowcount)
    # → 1  (one row deleted)

WHERE Conditions — The Full Reference

The .where() clause accepts condition expressions built from column objects. You can combine multiple conditions using and_(), or_(), and not_(). Every operator you know from SQL has a Python equivalent.

8.1 — Basic comparison operators
users.c.age == 28
WHERE users.age = 28
Equals. Note: Python uses == (double equals), not =.
users.c.age != 28
WHERE users.age != 28
Not equal to. Returns all rows where age is anything except 28.
users.c.age > 25
WHERE users.age > 25
Greater than. Works with numbers, dates, and strings (alphabetical).
users.c.age >= 25
WHERE users.age >= 25
Greater than or equal to. Includes the value 25 itself.
users.c.age < 30
WHERE users.age < 30
Less than. Does not include the value 30.
users.c.age <= 30
WHERE users.age <= 30
Less than or equal to. Includes the value 30.
8.2 — Combining multiple conditions with and_, or_, not_
and_() — ALL conditions must be true
from sqlalchemy import and_, or_, not_

# Users aged 25–35 AND from New York
stmt = select(users).where(
    and_(
        users.c.age >= 25,
        users.c.age <= 35,
        users.c.city == "New York"
    )
)
# SQL: WHERE age >= 25 AND age <= 35
#            AND city = 'New York'
or_() — AT LEAST ONE condition must be true
# Users from London OR Tokyo
stmt = select(users).where(
    or_(
        users.c.city == "London",
        users.c.city == "Tokyo"
    )
)
# SQL: WHERE city = 'London' OR city = 'Tokyo'
not_() — REVERSES a condition
# Everyone who is NOT from Paris
stmt = select(users).where(
    not_(users.c.city == "Paris")
)
# SQL: WHERE NOT (city = 'Paris')
NESTING — and_ inside or_
# (New York AND age>25) OR (London AND age>30)
stmt = select(users).where(
    or_(
        and_(
            users.c.city == "New York",
            users.c.age > 25
        ),
        and_(
            users.c.city == "London",
            users.c.age > 30
        )
    )
)
8.3 — Special operators: IN, BETWEEN, LIKE, IS NULL
ALL SPECIAL WHERE OPERATORS
# IN — value is in a list
stmt = select(users).where(
    users.c.city.in_(["London", "Tokyo", "Paris"])
)
# SQL: WHERE city IN ('London', 'Tokyo', 'Paris')

# NOT IN — value is NOT in a list
stmt = select(users).where(
    users.c.city.not_in(["London", "Tokyo"])
)
# SQL: WHERE city NOT IN ('London', 'Tokyo')

# BETWEEN — value is within a range (inclusive)
stmt = select(users).where(
    users.c.age.between(25, 32)
)
# SQL: WHERE age BETWEEN 25 AND 32

# LIKE — pattern matching (% = any characters, _ = one character)
stmt = select(users).where(
    users.c.name.like("A%")          # names starting with A
)
# SQL: WHERE name LIKE 'A%'

# ILIKE — same as LIKE but case-insensitive (PostgreSQL)
stmt = select(users).where(
    users.c.name.ilike("%alice%")   # case-insensitive contains
)
# SQL: WHERE name ILIKE '%alice%'

# IS NULL — column has no value (the Python None)
stmt = select(users).where(
    users.c.city.is_(None)
)
# SQL: WHERE city IS NULL

# IS NOT NULL — column has a value
stmt = select(users).where(
    users.c.city.is_not(None)
)
# SQL: WHERE city IS NOT NULL
Why in_() not in()?

in is a reserved keyword in Python (used in for x in list), so SQLAlchemy uses in_() with an underscore to avoid the conflict. The same applies to not_in() and is_().

9.1 — INNER JOIN (default join)
INNER JOIN returns rows that have a match in both tables. If a user has no orders, they are excluded. Use .join(right_table, condition) chained onto select().
Python
stmt = (
    select(

ORDER BY lets you control the order rows come back in. LIMIT restricts how many rows you get, and OFFSET lets you skip rows — together they power pagination ("give me page 2 of results").

ORDERING ROWS

stmt = select(users).order_by(

stmt = select(users).order_by(
    asc(users.c.city),
    desc(users.c.age)

GROUP BY & Aggregates — Summarizing Data

GROUP BY

GROUP BY collects rows that share the same value in a column and groups them together so you can calculate summaries — like counting how many orders each user has, or finding the total amount spent per city.

Aggregate functions (func.count, func.sum, func.avg, etc.) perform the calculation across each group. You import func from SQLAlchemy to use them.

11.1 — Aggregate functions reference
func.count()
Counts the number of rows (or non-NULL values in a column) in each group.
func.count(orders.c.id)
→ SQL: count(orders.id)
func.sum()
Adds up all values in a column across the group. Used for totals.
func.sum(orders.c.amount)
→ SQL: sum(orders.amount)
func.avg()
Computes the arithmetic average (mean) of values in a column.
func.avg(users.c.age)
→ SQL: avg(users.age)
func.min()
Returns the smallest value in a column within the group.
func.min(orders.c.amount)
→ SQL: min(orders.amount)
func.max()
Returns the largest value in a column within the group.
func.max(orders.c.amount)
→ SQL: max(orders.amount)
func.count(*)
Counts ALL rows including NULLs. Pass the literal(1) or use func.count().
func.count()
→ SQL: count(*)

Executing Queries & Transactions

Every query you build needs to be executed — sent to the database, run, and the result returned. In SQLAlchemy Core, this is done through a Connection object obtained from the engine. There are two ways to get a connection, and choosing the right one matters.

12.1 — engine.connect() vs engine.begin()
engine.connect()
For reading data (SELECT)
Opens a connection but does not start an automatic transaction. Changes are NOT committed unless you call conn.commit() manually. Best used for read-only operations.
READ PATTERN
with engine.connect() as conn:
    result = conn.execute(
        select(users)
    )
    rows = result.fetchall()
# conn auto-closed when block exits
engine.begin()
For writing data (INSERT / UPDATE / DELETE)
Opens a connection AND starts a transaction immediately. When the with block exits successfully, it auto-commits. If an error is raised, it auto-rollbacks. Use this for all writes.
WRITE PATTERN ✓
with engine.begin() as conn:
    conn.execute(
        insert(users).values(name="Alice")
    )
# auto-commits here — Alice is saved ✓
# if error → auto-rollback, Alice NOT saved
12.2 — What is a transaction, and why does it matter?

A transaction is a group of database operations that either all succeed together or all fail together. Think of it like this: you are moving money from Account A to Account B. That involves two steps — subtract from A, add to B. If step 1 succeeds but step 2 fails, you have lost money. A transaction prevents this — both steps happen or neither does.

In SQLAlchemy Core, when you use engine.begin(), you are inside a transaction. All your INSERT, UPDATE, and DELETE calls are part of that transaction. At the end of the with block, if no error occurred, the transaction is committed (all changes saved permanently). If any error was raised, the transaction is rolled back (all changes discarded, as if nothing happened).

TRANSACTION — all or nothing
# Transfer $100 from user 1 to user 2
# Both operations must succeed or both must fail

with engine.begin() as conn:
    # Step 1: Subtract from user 1's balance
    conn.execute(
        update(accounts)
        .where(accounts.c.user_id == 1)
        .values(balance=accounts.c.balance - 100)
    )

    # Step 2: Add to user 2's balance
    conn.execute(
        update(accounts)
        .where(accounts.c.user_id == 2)
        .values(balance=accounts.c.balance + 100)
    )

    # If we reach here without error → COMMIT
    # Both updates are saved. Transfer complete. ✓

    # If an error happens on Step 2 → ROLLBACK
    # Step 1 is also undone. No money lost. ✓
12.3 — Manual commit and rollback using engine.connect()
MANUAL TRANSACTION CONTROL
# When you need explicit control over commit/rollback
with engine.connect() as conn:
    try:
        conn.execute(insert(users).values(name="Test"))
        conn.execute(insert(orders).values(user_id=99, product="Item"))
        conn.commit()          # ← explicitly save changes
        print("Saved!")
    except Exception as e:
        conn.rollback()         # ← explicitly undo changes
        print(f"Failed, rolled back: {e}")
Recommendation: use engine.begin() for writes

For almost all cases, engine.begin() is the cleaner pattern — it handles commit and rollback automatically. Use engine.connect() with manual commit/rollback only when you need fine-grained control over when exactly the commit happens (for example, inside a long-running batch job).
