Loading Strategies — Controlling When Related Data is Fetched

When you access a relationship like user.orders, SQLAlchemy needs to fetch those related rows from the database. How and when it does that is controlled by the loading strategy.

Choosing the wrong strategy is one of the most common causes of slow SQLAlchemy applications. Too lazy = many tiny queries. Too eager = huge JOINs you didn't need. Understanding each option lets you match the strategy to the situation.

There are 5 strategies: lazy (default), selectin (recommended for lists), joined (best for filters), subquery (legacy), and raise (for catching accidental lazy loads in production).

The golden rule

The default is lazy — each relationship fires a new query the first time you access it. This is fine for 1-2 objects but catastrophic in a loop. Always use selectinload() or joinedload() when you know you will access the relationship for multiple objects.

Lazy lazy="select"
The default. Fires a new SELECT the first time you access the attribute. 0 extra queries if you never touch it, but 1 query per object if you loop.
relationship("Order", lazy="select")
N queries in a loop ⚠
Selectin selectinload()
Recommended for collections. Runs 1 extra SELECT using WHERE id IN (...). Loads all children in one go — 2 queries total regardless of how many parents.
.options(selectinload(User.orders))
2 queries total ✓
Joined joinedload()
Uses a SQL JOIN in the same query. 1 query total. Best when you need to filter by the related model's columns. Can duplicate parent rows — use .unique().
.options(joinedload(User.orders))
1 query total ✓
Raise lazy="raise"
Raises an error if you accidentally access the relationship after the session is closed. Use in production to catch lazy-load bugs before they hit users.
relationship("Order", lazy="raise")
Catches bugs ✓
The N+1 Problem — visualised step by step
What happens when you use lazy loading in a loop with 4 users

LAZY vs SELECTIN vs JOINED
from sqlalchemy.orm import selectinload, joinedload

with Session(engine) as session:

    # ── Lazy (default) ── fires 1 query per user!
    users = session.execute(select(User)).scalars().all()
    for u in users:
        print(u.orders)   # ← SELECT fired here!

    # ── selectinload ── 2 queries total, always ──
    users = session.execute(
        select(User).options(selectinload(User.orders))
    ).scalars().all()
    for u in users:
        print(u.orders)   # ← already loaded, no query

    # ── joinedload ── 1 query with a LEFT JOIN ──
    users = session.execute(
        select(User).options(joinedload(User.orders))
    ).unique().scalars().all()
    # .unique() needed because JOIN duplicates parent rows

    # ── Nested eager loading (orders AND their items) ──
    users = session.execute(
        select(User).options(
            selectinload(User.orders).selectinload(Order.items)
        )
    ).scalars().all()
SETTING DEFAULT LOADING IN THE MODEL
from sqlalchemy.orm import relationship

class User(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(primary_key=True)

    # Default lazy (fires a query each time)
    orders_lazy: Mapped[List["Order"]] = relationship(
        back_populates="user",
        lazy="select"          # this is the default
    )

    # Always eager-load with selectin
    orders_eager: Mapped[List["Order"]] = relationship(
        back_populates="user",
        lazy="selectin"        # always selectin-loaded
    )

    # Raise error if accidentally accessed lazily
    orders_strict: Mapped[List["Order"]] = relationship(
        back_populates="user",
        lazy="raise"           # prevents accidental lazy loads
    )

    # Write-only — can add but never read (large collections)
    orders_write: Mapped[List["Order"]] = relationship(
        back_populates="user",
        lazy="write_only"      # only for .add() and .remove()
    )
When to set default loading in the model

Set lazy="selectin" on a relationship when you always need the children (e.g. every user needs their role). Prefer query-time loading for everything else — it gives you flexibility.

Cascade — what happens to children when the parent changes

The cascade option controls what happens to related objects when you perform an action on the parent. By default, saving a parent auto-saves new children you've appended (this is called the save-update cascade). But you can also make SQLAlchemy auto-delete children when you delete the parent — this is cascade="all, delete-orphan".

Think of it like this: if you delete a User row, do you want their Order rows deleted too? Or do you want the database to refuse the delete because orders still reference the user? Cascade decides this in Python; the foreign key ondelete decides it in the database.

CASCADE OPTIONS
class User(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(primary_key=True)

    # DEFAULT: save-update, merge
    # Appending a new order auto-saves it with the user
    orders_default: ... = relationship("Order")

    # delete-orphan: if order removed from list → deleted
    orders_strict: ... = relationship(
        "Order",
        cascade="all, delete-orphan"
    )

# With delete-orphan:
with Session(engine) as session:
    user = session.get(User, 1)

    # Deleting the user also deletes all their orders
    session.delete(user)
    session.commit()
    # DELETE FROM orders WHERE user_id = 1
    # DELETE FROM users WHERE id = 1

    # Removing from list deletes the order row
    user.orders.remove(user.orders[0])
    session.commit()
    # DELETE FROM orders WHERE id = <that order's id>

Association Objects — Many-to-Many with Extra Data

In Module 05 we used a plain Table for many-to-many relationships. That works when the join table only has the two foreign key columns. But what if you need to store extra information about the connection itself?

For example: a Student can enrol in many Courses, and a Course can have many Students. But you also want to store when the student enrolled and what grade they got. Those pieces of data belong on the join — they are about the connection, not the student or the course alone.

When the join table has extra columns, you model it as a full Python class — called an association object. This is just a normal model that has two foreign keys plus any extra columns you need.

DEFINING THE ASSOCIATION OBJECT
from datetime import date
from sqlalchemy import String, Date, ForeignKey
from sqlalchemy.orm import (
    Mapped, mapped_column, relationship, DeclarativeBase
)
from typing import List, Optional

class Base(DeclarativeBase): pass


class Enrolment(Base):
    "The association object — the join table as a full model"
    __tablename__ = "enrolments"

    # Two foreign keys — the join
    student_id: Mapped[int] = mapped_column(
        ForeignKey("students.id"), primary_key=True
    )
    course_id: Mapped[int] = mapped_column(
        ForeignKey("courses.id"), primary_key=True
    )

    # Extra data about the connection itself
    enrolled_on: Mapped[date] = mapped_column(Date)
    grade: Mapped[Optional[str]] = mapped_column(String(5))

    # Relationships back to both sides
    student: Mapped["Student"] = relationship(back_populates="enrolments")
    course:  Mapped["Course"]  = relationship(back_populates="enrolments")


class Student(Base):
    __tablename__ = "students"
    id:   Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(100))
    enrolments: Mapped[List[Enrolment]] = relationship(
        back_populates="student"
    )


class Course(Base):
    __tablename__ = "courses"
    id:    Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str] = mapped_column(String(200))
    enrolments: Mapped[List[Enrolment]] = relationship(
        back_populates="course"
    )
USING THE ASSOCIATION OBJECT
with Session(engine) as session:

    alice  = Student(name="Alice")
    python = Course(title="Python 101")
    session.add_all([alice, python])
    session.commit()

    # Enrol Alice in the course with extra data
    enrolment = Enrolment(
        student_id  = alice.id,
        course_id   = python.id,
        enrolled_on = date(2024, 9, 1),
        grade       = None       # not graded yet
    )
    session.add(enrolment)
    session.commit()

    # Later: set Alice's grade
    enr = session.execute(
        select(Enrolment).where(
            Enrolment.student_id == alice.id,
            Enrolment.course_id  == python.id
        )
    ).scalars().first()
    enr.grade = "A"
    session.commit()

    # See all courses Alice is in + her grades
    for e in alice.enrolments:
        print(e.course.title, e.grade, e.enrolled_on)
Association object vs plain table

Use a plain Table (Module 05 style) when the join table only has the two foreign keys. Switch to an association object the moment you need any extra column on the join — enrolled_on, grade, quantity, role, status, priority…

Self-Referential Relationships — Trees and Hierarchies

Sometimes rows in a table relate to other rows in the same table. This is called a self-referential relationship. Classic examples: a category tree (Electronics → Phones → Smartphones), an employee org chart (manager → employee), or threaded comments (comment → replies).

The pattern is the same as a regular one-to-many relationship, except the ForeignKey points back to the same table instead of a different one. The relationship also points to the same model class — written as a string to avoid circular reference issues.

Each row has a parent_id that points to the id of another row in the same table. Root nodes have parent_id = NULL.
SELF-REFERENTIAL MODEL
class Category(Base):
    __tablename__ = "categories"

    id:   Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(100))

    # FK points to the SAME TABLE's id column
    parent_id: Mapped[Optional[int]] = mapped_column(
        ForeignKey("categories.id")
    )

    # Relationship to children (many)
    children: Mapped[List["Category"]] = relationship(
        back_populates="parent",
        # remote_side tells SA which side is "one"
    )

    # Relationship back to the single parent
    parent: Mapped[Optional["Category"]] = relationship(
        back_populates="children",
        remote_side="Category.id"
    )

    def __repr__(self):
        return f"<Category {self.name!r}>"
USING THE TREE
with Session(engine) as session:

    # Build the tree
    electronics = Category(name="Electronics")
    phones      = Category(name="Phones")
    smartphones = Category(name="Smartphones")

    # Set the parent-child connections
    phones.parent      = electronics
    smartphones.parent = phones

    session.add(electronics)   # cascade saves children too
    session.commit()

    # Navigate the tree
    root = session.get(Category, electronics.id)
    print(root.children)
    # → [<Category 'Phones'>]

    print(root.children[0].children)
    # → [<Category 'Smartphones'>]

    print(smartphones.parent.name)
    # → "Phones"

    # Find all root categories (no parent)
    roots = session.execute(
        select(Category).where(
            Category.parent_id == None
        )
    ).scalars().all()
    # → [<Category 'Electronics'>]

Inheritance Mapping — Storing Python Subclasses in the Database

Python supports class inheritance — you can have a Vehicle base class and subclasses like Car, Truck, and Motorcycle. Each subclass shares the common fields from Vehicle but adds its own specific fields.

SQLAlchemy gives you three strategies to map this inheritance structure to a relational database. Each makes a different trade-off between simplicity and query performance.

Strategy 1
Single Table Inheritance
All classes — parent and all subclasses — stored in one table. A special type column (called the discriminator) says which subclass each row is. Columns belonging to other subclasses are NULL for rows that don't use them.
✓Simplest to query — no JOINs
✓Fast reads and writes
✗Table gets wide — many NULL columns
✗Can't enforce NOT NULL on subclass cols
Best for: small, similar subclasses
Strategy 2
Joined Table Inheritance
The parent class has its own table with the shared fields. Each subclass has its own separate table for its specific fields. To get a full object, SQLAlchemy JOINs the two tables together.
✓Clean, normalised schema — no NULLs
✓Can enforce constraints per subclass
✗Every query requires a JOIN
✗Slightly more complex to write
Best for: subclasses with many unique fields
Strategy 3
Concrete Table Inheritance
Each subclass gets its own completely separate table with all columns (shared + unique) duplicated. No shared parent table. No JOINs. Querying across all subclasses requires a UNION.
✓Fastest for reading one subclass
✓No NULLs, no JOINs
✗Schema duplication across tables
✗Querying all types requires UNION
Best for: very different subtypes, rarely queried together

class Vehicle(Base):
    __tablename__ = "vehicles"

    id:    Mapped[int] = mapped_column(primary_key=True)
    make:  Mapped[str] = mapped_column(String(50))
    model: Mapped[str] = mapped_column(String(50))

    # This is the magic column — tells SQLAlchemy which subclass each row is
    type: Mapped[str] = mapped_column(String(20))

    __mapper_args__ = {
        "polymorphic_on": "type",    # which column is the discriminator
        "polymorphic_identity": "vehicle"  # value for base rows
    }


class Car(Vehicle):
    num_doors: Mapped[Optional[int]]

    __mapper_args__ = { "polymorphic_identity": "car" }
    # No __tablename__ — uses the parent's table ✓


class Truck(Vehicle):
    payload_kg: Mapped[Optional[int]]

    __mapper_args__ = { "polymorphic_identity": "truck" }


class Motorcycle(Vehicle):
    engine_cc: Mapped[Optional[int]]

    __mapper_args__ = { "polymorphic_identity": "moto" }

Hybrid Properties — Computed Values in Both Python and SQL

A hybrid property is a computed attribute that works in two different contexts:

1. On an object instance — it runs like a normal Python property: takes self, accesses other attributes, and returns a computed value. For example, user.full_name might return f"{self.first_name} {self.last_name}".

2. In a SQL query — it produces a SQL expression that the database can evaluate. So you can filter with .where(User.full_name == "Alice Smith") and SQLAlchemy translates it to SQL: WHERE first_name || ' ' || last_name = 'Alice Smith'.

Without hybrid properties, you would have to duplicate the logic — once as a Python property and once as a raw SQL expression. The hybrid property handles both with a single definition.

Same property — two contexts
🐍 PYTHON CONTEXT — accesses self
user = session.get(User, 1)
print(user.full_name)
# calls the @hybrid_property function
# → self.first_name + ' ' + self.last_name
# → "Alice Smith"
The regular Python property decorator runs and does Python string operations. No SQL involved.
🗄️ SQL CONTEXT — produces SQL expression
select(User).where(
  User.full_name == "Alice Smith"
)
-- calls the @full_name.expression function
WHERE first_name || ' ' || last_name = 'Alice Smith'
The expression variant kicks in and produces SQL. The database evaluates it — not Python.
5.1 — Full name property
HYBRID PROPERTY — full_name
from sqlalchemy.ext.hybrid import hybrid_property
from sqlalchemy import String

class User(Base):
    __tablename__ = "users"
    id:         Mapped[int] = mapped_column(primary_key=True)
    first_name: Mapped[str] = mapped_column(String(50))
    last_name:  Mapped[str] = mapped_column(String(50))
    age:        Mapped[int]

    # ── Python instance accessor ──
    @hybrid_property
    def full_name(self) -> str:
        return f"{self.first_name} {self.last_name}"

    # ── SQL expression accessor (used in queries) ──
    @full_name.expression
    @classmethod
    def full_name(cls):
        # Returns a SQL expression — database does the work
        return cls.first_name + " " + cls.last_name

    # ── Setter — allows user.full_name = "Bob Jones" ──
    @full_name.setter
    def full_name(self, value: str):
        parts = value.split(" ", 1)
        self.first_name = parts[0]
        self.last_name  = parts[1] if len(parts) > 1 else ""
USING THE HYBRID PROPERTY
with Session(engine) as session:

    # ── Python context: access on an object ──
    user = session.get(User, 1)
    print(user.full_name)
    # → "Alice Smith"  (Python string concatenation)

    # ── Setter splits the name ──
    user.full_name = "Alice Johnson"
    print(user.first_name)   # → "Alice"
    print(user.last_name)    # → "Johnson"
    session.commit()

    # ── SQL context: filter with it ──
    users = session.execute(
        select(User).where(
            User.full_name == "Alice Smith"
        )
    ).scalars().all()
    # SQL: WHERE first_name || ' ' || last_name = 'Alice Smith'

    # ── Filter with LIKE — partial name search ──
    users = session.execute(
        select(User).where(
            User.full_name.like("Alice%")
        )
    ).scalars().all()
    # SQL: WHERE (first_name || ' ' || last_name) LIKE 'Alice%'

class User(Base):
    __tablename__ = "users"
    id:  Mapped[int] = mapped_column(primary_key=True)
    age: Mapped[int]

    @hybrid_property
    def is_adult(self) -> bool:
        return self.age >= 18

    @is_adult.expression
    @classmethod
    def is_adult(cls):
        return cls.age >= 18   # SQL: age >= 18

# Python:
user = User(age=16)
print(user.is_adult)   # → False

# SQL: WHERE age >= 18
adults = session.execute(
    select(User).where(User.is_adult == True)
).scalars().all()
NUMERIC HYBRID — discounted price
class Product(Base):
    __tablename__ = "products"
    id:       Mapped[int]   = mapped_column(primary_key=True)
    price:    Mapped[float]
    discount: Mapped[float] # 0.0 to 1.0 (e.g. 0.2 = 20% off)

    @hybrid_property
    def sale_price(self) -> float:
        return self.price * (1 - self.discount)

    @sale_price.expression
    @classmethod
    def sale_price(cls):
        # SQL: price * (1 - discount)
        return cls.price * (1 - cls.discount)

# Python: access on an object
p = Product(price=100, discount=0.2)
print(p.sale_price)  # → 80.0

# SQL: find cheap products (after discount)
cheap = session.execute(
    select(Product).where(
        Product.sale_price < 50
    )
).scalars().all()
# SQL: WHERE price * (1 - discount) < 50

Custom Data Types — Storing Special Python Objects

Sometimes the data you want to store is not a simple string or number. You might want to store a Python dictionary as JSON, a comma-separated list as a Python list, an enum value, an encrypted string, or a currency amount stored as an integer to avoid floating-point errors.

SQLAlchemy's TypeDecorator lets you define a custom column type that automatically converts the value when reading from or writing to the database. You write once; it works everywhere.

The idea: you define process_bind_param (Python → database, called on INSERT/UPDATE) and process_result_value (database → Python, called on SELECT). SQLAlchemy calls these automatically.

6.1 — Storing a Python list as a comma-separated string
CUSTOM TYPE — CommaSeparatedList
from sqlalchemy import types

class CommaSeparatedList(types.TypeDecorator):
    """Store a Python list as 'a,b,c' in the database."""

    # The underlying database type (what column is in the DB)
    impl = types.String

    # Ensure strings are compared correctly
    cache_ok = True

    # Python → Database: called on INSERT / UPDATE
    def process_bind_param(self, value, dialect):
        if value is None:
            return None
        return ",".join(value)    # ["a","b"] → "a,b"

    # Database → Python: called on SELECT
    def process_result_value(self, value, dialect):
        if not value:
            return []
        return value.split(",")   # "a,b" → ["a","b"]


# Use it in a model just like any built-in type
class Article(Base):
    __tablename__ = "articles"
    id:    Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str] = mapped_column(String(200))

    # Tags stored as "python,orm,sqlalchemy" in DB
    # but returned as ["python", "orm", "sqlalchemy"] in Python
    tags: Mapped[list] = mapped_column(CommaSeparatedList())
USING IT — seamless Python lists
with Session(engine) as session:

    # Write — give it a Python list
    article = Article(
        title = "SQLAlchemy Guide",
        tags  = ["python", "orm", "database"]
    )
    session.add(article)
    session.commit()
    # DB stores: tags = "python,orm,database"

    # Read — get back a Python list
    a = session.get(Article, article.id)
    print(type(a.tags))   # → <class 'list'>
    print(a.tags)         # → ['python', 'orm', 'database']

    # Add a tag
    a.tags.append("tutorial")
    session.commit()
    # DB now stores: "python,orm,database,tutorial"

    # Verify it came back correctly
    fresh = session.get(Article, article.id)
    print("tutorial" in fresh.tags)  # → True ✓

6.2 — Storing JSON / dicts (built-in + custom)

SQLAlchemy comes with a built-in JSON type that handles Python dicts and lists automatically. For databases that don't support native JSON (older SQLite), you can use TypeDecorator to serialise to a text column.

The built-in JSON is the recommended choice for PostgreSQL (which has a native jsonb type), MySQL, and modern SQLite. It uses the database's native JSON support when available, falling back to text otherwise.

JSON TYPE — built-in and custom
import json
from sqlalchemy import JSON, types

# ── BUILT-IN JSON type (PostgreSQL / MySQL / SQLite 3.38+)
class UserProfile(Base):
    __tablename__ = "user_profiles"
    id:       Mapped[int]  = mapped_column(primary_key=True)
    username: Mapped[str]  = mapped_column(String(100))
    metadata: Mapped[dict] = mapped_column(JSON)
    # stores dicts directly — Python dict in, dict out

# Usage:
with Session(engine) as session:
    profile = UserProfile(
        username="alice",
        metadata={"theme": "dark", "lang": "en", "notifs": True}
    )
    session.add(profile)
    session.commit()

    p = session.get(UserProfile, profile.id)
    print(p.metadata["theme"])   # → "dark"

    # PostgreSQL JSON path query
    dark_users = session.execute(
        select(UserProfile).where(
            UserProfile.metadata["theme"].as_string() == "dark"
        )
    ).scalars().all()
6.3 — Python Enum types — type-safe status columns

If a column can only hold a fixed set of values (like "pending", "shipped", "delivered") you should use Python's enum.Enum class instead of plain strings. This gives you:

1. Auto-complete — your IDE knows the valid values.
2. Type safety — you can't accidentally set "shiped" (typo).
3. Cleaner code — compare with OrderStatus.SHIPPED not "shipped".

SQLAlchemy's Enum type maps directly to a database ENUM (PostgreSQL) or a CHECK constraint (SQLite/MySQL).

ENUM TYPE
import enum
from sqlalchemy import Enum as SAEnum

# 1. Define the Python enum
class OrderStatus(enum.Enum):
    PENDING   = "pending"
    SHIPPED   = "shipped"
    DELIVERED = "delivered"
    CANCELLED = "cancelled"


# 2. Use it in a model
class Order(Base):
    __tablename__ = "orders"
    id:      Mapped[int]         = mapped_column(primary_key=True)
    product: Mapped[str]
    status:  Mapped[OrderStatus]  = mapped_column(
        SAEnum(OrderStatus),
        default=OrderStatus.PENDING
    )


# 3. Use it in code — type-safe ✓
with Session(engine) as session:
    order = Order(product="Laptop")
    session.add(order)
    session.commit()
    print(order.status)          # → OrderStatus.PENDING

    order.status = OrderStatus.SHIPPED
    session.commit()

    # Filter by enum — no magic strings
    shipped = session.execute(
        select(Order).where(
            Order.status == OrderStatus.SHIPPED
        )
    ).scalars().all()
6.4 — Encrypting sensitive data with TypeDecorator

The same TypeDecorator pattern can be used to encrypt data before storing and decrypt it when reading. You write a custom type that encrypts in process_bind_param and decrypts in process_result_value. The rest of your code works with plain strings — the encryption is completely transparent.

This is useful for storing things like API keys, tokens, or personal data that you want encrypted at rest in the database. Even if someone reads the raw database, they only see ciphertext.

TypeDecorator summary — the pattern

1. impl — pick the underlying DB column type (String, Integer, etc.)
2. process_bind_param — Python → DB (called on write)
3. process_result_value — DB → Python (called on read)
4. cache_ok = True — always set this for correctness

ENCRYPTED TYPE — with Fernet (cryptography library)
from cryptography.fernet import Fernet
from sqlalchemy import types

# Generate a key once and store it securely (env var)
SECRET_KEY = Fernet.generate_key()
fernet = Fernet(SECRET_KEY)


class EncryptedString(types.TypeDecorator):
    """Encrypt on write, decrypt on read — transparent to the model."""
    impl = types.LargeBinary   # store raw bytes
    cache_ok = True

    def process_bind_param(self, value, dialect):
        if value is None: return None
        return fernet.encrypt(value.encode())

    def process_result_value(self, value, dialect):
        if value is None: return None
        return fernet.decrypt(value).decode()


class PaymentInfo(Base):
    __tablename__ = "payment_info"
    id:          Mapped[int] = mapped_column(primary_key=True)
    user_id:     Mapped[int]
    card_number: Mapped[str] = mapped_column(EncryptedString())

# Usage — works exactly like a plain string field
with Session(engine) as session:
    info = PaymentInfo(user_id=1, card_number="4111111111111111")
    session.add(info)
    session.commit()
    # DB stores: encrypted bytes — unreadable without the key

    loaded = session.get(PaymentInfo, info.id)
    print(loaded.card_number)  # → "4111111111111111" (decrypted) ✓
Module Summary — What you learned
Loading Strategies
Lazy is the default but causes N+1 in loops. Use selectinload() for collections (2 queries total). Use joinedload() when filtering on the related model. Use lazy="raise" to catch bugs.
Association Objects
When a many-to-many join table needs extra columns (dates, grades, status), model the join table as a full Python class. Navigate via the association class, not directly.
Self-referential
A FK that points to the same table's PK. Used for trees (categories), org charts (employees), threaded comments. Use remote_side to tell SQLAlchemy which end is "one".
Inheritance Mapping
STI = one table + discriminator column (simple, wide table). JTI = parent table + subclass tables + JOINs (clean schema). CTI = fully separate tables per subclass (fastest reads, no shared table).
Hybrid Properties
Use @hybrid_property + @x.expression for computed values that work both on Python objects (returns value) and in SQL WHERE clauses (returns SQL expression). Add @x.setter to allow writes.
Custom Types
Subclass TypeDecorator, set impl, write process_bind_param (Python→DB) and process_result_value (DB→Python). Works for lists, JSON, enums, encryption — anything.

Mixins — Write Column Logic Once, Use It on Every Model

Look at any real production codebase and you'll see the same columns on almost every table: id, created_at, updated_at. Copying those three fields into every model is repetitive and error-prone — if you ever want to change the timestamp default, you'd have to touch 20 files.

A mixin is a plain Python class that holds column definitions but is not itself a database table. You inherit from it in your models, and those columns are automatically added. It's the same idea as a reusable component.

Mixins are not a SQLAlchemy-specific concept — they're standard Python multiple inheritance. SQLAlchemy just happens to understand them when combined with DeclarativeBase.

One change, all models update

If you put created_at in a mixin and later decide every timestamp should use timezone=True, you change it in one place and every model that inherits the mixin gets the fix automatically.

DEFINING MIXINS — mixins.py
import uuid
from datetime import datetime
from uuid import UUID
from sqlalchemy import func
from sqlalchemy.orm import Mapped, mapped_column


class UUIDPrimaryKeyMixin:
    """Adds a UUID primary key to any model."""
    id: Mapped[UUID] = mapped_column(
        primary_key=True,
        default=uuid.uuid4,
        nullable=False
    )


class TimestampMixin:
    """Adds created_at and updated_at to any model."""
    created_at: Mapped[datetime] = mapped_column(
        default=datetime.utcnow,
        server_default=func.now(),   # DB enforces it too
        nullable=False
    )
    updated_at: Mapped[datetime] = mapped_column(
        default=datetime.utcnow,
        server_default=func.now(),
        onupdate=datetime.utcnow,     # auto-update on every UPDATE
        nullable=False
    )
USING MIXINS IN YOUR MODELS
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column
from myapp.mixins import UUIDPrimaryKeyMixin, TimestampMixin

class Base(DeclarativeBase): pass


# Inherit from BOTH mixins + Base
class User(UUIDPrimaryKeyMixin, TimestampMixin, Base):
    __tablename__ = "users"
    name:  Mapped[str]
    email: Mapped[str]
    # User automatically gets: id, created_at, updated_at
    # Final columns: id | name | email | created_at | updated_at


class Product(UUIDPrimaryKeyMixin, TimestampMixin, Base):
    __tablename__ = "products"
    name:  Mapped[str]
    price: Mapped[float]
    # Final columns: id | name | price | created_at | updated_at


# ── Using the mixin columns ──
user = User(name="Alice", email="alice@example.com")
print(user.id)         # UUID already set — no DB call needed
print(user.created_at) # datetime already set

session.add(user)
session.commit()
# The mixin fields are treated exactly like regular columns in queries
from sqlalchemy import select
recent = session.scalars(
    select(User).order_by(User.created_at.desc())
)
Mixin vs Abstract Base

A mixin is NOT a table — it has no __tablename__. An abstract base (with __abstract__ = True) is similar but is an actual SQLAlchemy base class. For shared columns, mixins are simpler and more flexible.

Beyond timestamps — other useful mixins
SoftDeleteMixin
Adds is_deleted and deleted_at. Instead of deleting a row, you set is_deleted = True. The row stays in the DB for audit history.
SlugMixin
Adds a slug column used in URLs (e.g. /blog/my-post-title). Useful for any model that needs human-readable URL paths.
AuditMixin
Adds created_by and updated_by user IDs. Tracks who made each change — useful for admin panels and compliance.
